# DESi v1.3 — Logical Consilium / Bridge Review

<!-- [legacy-unanchored] — pre-v2.0 prose report; no JSON artefact exists for this version. -->

v1.3 is **the first authority-free bridge consilium**. v1.2 turned
the gap audit into a belief gate; it could *propose* bridge claims
but never accept them. v1.3 introduces the adversarial review body
that decides — under hard constraints — whether a bridge claim
earns ``LOGICALLY_SUPPORTED``.

Design principle (hard invariant):

> A bridge claim is never accepted because it is convenient,
> intuitive, popular, repeated, highly cited, authored by experts,
> or supported by institutional prestige.
>
> A bridge claim may only be accepted if an adversarial consilium
> fails to find an unresolved logical gap. A single unresolved veto
> blocks acceptance.

## The four roles — closed enum

```python
class ConsiliumRole(str, Enum):
    LOGICIAN         = "logician"
    SKEPTIC          = "skeptic"
    DOMAIN_EXAMINER  = "domain_examiner"
    INTEGRATOR       = "integrator"
```

| Role             | Asks                                                |
|---               |---                                                   |
| LOGICIAN         | Does the bridge structurally connect a premise to the conclusion? |
| SKEPTIC          | Is there an adversarial condition that negates the bridge?         |
| DOMAIN_EXAMINER  | Is the vocabulary metaphorical in the declared context?            |
| INTEGRATOR       | Does the bridge mention the conclusion's subject AND share a content lemma with a premise? |

Reviews are independent — no role reads another role's output.
Aggregation is **set-based**, so the verdict is invariant under
role-order permutation (INV-C1).

## The four verdicts — closed enum

```python
class Verdict(str, Enum):
    ACCEPT_AS_BRIDGE     = "accept_as_bridge"
    REJECT_BRIDGE        = "reject_bridge"
    NEEDS_MORE_PREMISES   = "needs_more_premises"
    VETO                  = "veto"
```

Aggregation rules (set-based, evaluated in this priority):

1. **REJECT_BRIDGE** — input contract failure (missing bridge_id,
   text, source_claim_id, rationale, or original_text). No roles
   run. Hard pre-empts everything.
2. **VETO** — any role has `unresolved_gap=True`. Even one. No
   majority override (INV-C3).
3. **NEEDS_MORE_PREMISES** — no unresolved gap, but at least one
   role has `needs_more_premises=True` (DOMAIN_EXAMINER, today).
4. **ACCEPT_AS_BRIDGE** — all four passes clean.

## The four bridge states

```python
class BridgeState(str, Enum):
    UNDER_CONSILIUM      = "under_consilium"
    CONSILIUM_ACCEPTED   = "consilium_accepted"
    CONSILIUM_REJECTED   = "consilium_rejected"
    CONSILIUM_BLOCKED    = "consilium_blocked"
```

Mapping:

| Verdict                | Bridge state          |
|---                     |---                    |
| ACCEPT_AS_BRIDGE       | CONSILIUM_ACCEPTED    |
| REJECT_BRIDGE          | CONSILIUM_REJECTED    |
| VETO                   | CONSILIUM_BLOCKED     |
| NEEDS_MORE_PREMISES    | CONSILIUM_BLOCKED     |

## The five hard invariants

| ID    | Statement                                                               | Test                                                              |
|---    |---                                                                       |---                                                                 |
| INV-C1 | Changing role order must not change verdict                              | `test_all_24_permutations_yield_identical_verdict`                 |
| INV-C2 | Changing metadata must not change verdict                                | `test_c4_prestige_trap_same_verdict_across_authors`, `test_inv_c2_*` |
| INV-C3 | One unresolved veto must always block                                    | `test_single_counterexample_triggers_veto`, `test_veto_overrides_needs_more_premises` |
| INV-C4 | Accepted bridge replay_hash must be deterministic                        | `test_replay_hash_is_deterministic`, `test_replay_hash_independent_of_wall_clock` |
| INV-C5 | Same bridge text + same premises must always produce identical verdict   | `test_same_bridge_text_same_premises_same_verdict`                 |

The `BridgeConsilium.deliberate(..., source_metadata=None)`
signature accepts `source_metadata` for API parity with the v1.2
auditor, then immediately drops it (`del source_metadata`). The
authority-invariance tests pass Nobel-laureate / h_index=9999 /
citation_count=99999 metadata bombs; the verdict does not move.

```python
ConsiliumReplay(

    premise_ids=("pr_…", "pr_…"),

```

Hash inputs (in order, with NULL separators):

* normalised bridge_text (lowercase, whitespace-collapsed)
* source_claim_id
* verdict value
* SORTED premise_ids
* SORTED + normalised conditions

NOT in the hash: author, title, source, citation_count, role order,
timestamp, claim_id of the consilium result.

## The five required scenarios

### C1 — Rain street → ACCEPT_AS_BRIDGE

```
Original: "It is raining. Therefore the street is wet."
Bridge:   "the street is exposed to the rain"

  logician        ✓ (bridge connects "rain"→"rain" and shares "street" with conclusion)
  skeptic         ✓ (no additional conditions)
  domain_examiner ✓ (no context declared)
  integrator      ✓ (bridge mentions subject "street", links via "rain")

  verdict       = ACCEPT_AS_BRIDGE
  bridge_state  = CONSILIUM_ACCEPTED
  replay_hash   = cr_665fb456581e96a3
```

### C2 — Roof counterexample → VETO

```
Same bridge as C1, additional_conditions=("the street has a roof",)

  logician        ✓
  skeptic         ✗ unresolved_gap=True
                    counterexample: bridge claims 'exposed to the rain'
                    but condition 'the street has a roof' negates it
  domain_examiner ✓
  integrator      ✓

  verdict          = VETO
  bridge_state     = CONSILIUM_BLOCKED
  blocking_roles   = (SKEPTIC,)
```

### C3 — Metaphor trap → NEEDS_MORE_PREMISES

```
Original: "The market is hot. Therefore the city is flooded."
context = "financial_newspaper"

  logician         ✓
  skeptic          ✓
  domain_examiner  needs_more_premises=True
                   "'flooded' is ambiguous in context 'financial_newspaper':
                    in financial discourse 'flooded' typically means
                    saturated supply / capital inflow, not water."
  integrator       ✓

  verdict          = NEEDS_MORE_PREMISES
  bridge_state     = CONSILIUM_BLOCKED
```

### C4 — Multi-source prestige trap → identical verdict

The same off-topic bridge, proposed by:

* Nobel laureate
* university institute
* anonymous blogger
* alice with h_index=9999 and citation_count=99999

All four verdicts identical. Test:
`test_c4_prestige_trap_same_verdict_across_authors`.

### C5 — Role permutation → identical verdict + identical hash

```
24 permutations of (LOGICIAN, SKEPTIC, DOMAIN_EXAMINER, INTEGRATOR)
on the C1 inputs:

  distinct verdicts = 1   → {ACCEPT_AS_BRIDGE}
  distinct hashes   = 1   → {cr_665fb456581e96a3}
```

## Bridge promotion + claim upgrade

When the consilium accepts:

```python
new_bridge_claim = promote_accepted_bridge(bridge, verdict)
  # new_bridge_claim.state  == ClaimState.LOGICALLY_SUPPORTED
  # new_bridge_claim.method == "logical_bridge"   ← never rewritten
```

To upgrade the *original* claim (the one whose v1.2 audit returned
``BRIDGE_REQUIRED``):

```python
upgraded = upgrade_original_claim(
    original,
    accepted_bridge_state=BridgeState.CONSILIUM_ACCEPTED,
    verdict=Verdict.ACCEPT_AS_BRIDGE,
)
  # upgraded.state == ClaimState.LOGICALLY_SUPPORTED
```

Preconditions are enforced — any precondition failure raises
:class:`ClaimUpgradeError`:

* original.state must be ``BRIDGE_REQUIRED``
* bridge_state must be ``CONSILIUM_ACCEPTED``
* verdict must be ``ACCEPT_AS_BRIDGE``

The original's ``method`` and ``content`` are preserved verbatim;
``operator_path`` is extended with ``"consilium_upgrade"`` so the
audit trail names the upgrade event.

## What v1.3 deliberately does NOT do

* no prestige scoring
* no majority voting
* no confidence averaging
* no LLM authority
* no hidden heuristics
* no auto-upgrade of the original claim without explicit invocation
* no second activated config knob
* no Memory reads during a run
* no parallel mutations
* no Neo4j ledger backend

The directive: **DESi does not trust bridges. DESi cross-examines
them.**

## Open design questions for v1.4

1. **Counterexample-library growth.** v1.3 ships a deliberately small
   pair set (`exposed_to ↔ {roof, sheltered, covered}` etc.). v1.4
   needs a controlled growth policy — every new pattern is a code
   edit, must come with at least one direct test, and must declare
   the symmetry/transitivity contract of the negation.
2. **Multiple metaphor contexts on one document.** Today the
   DOMAIN_EXAMINER consumes one `context` string. Real documents may
   mix discourses (e.g. a science journalism piece). v1.4 should
   formalise multi-context evaluation while preserving INV-C2 (still
   no authority signal).
3. **Adversarial-input budget.** The consilium accepts an unbounded
   tuple of `additional_conditions`. v1.4 should cap the input size
   so a single call cannot run the SKEPTIC against thousands of
   prepared negators; the cap itself must be auditable.
4. **Cross-bridge follow-up.** When a bridge is BLOCKED with
   NEEDS_MORE_PREMISES, the caller might offer a second bridge to
   close the metaphor gap. v1.4 needs a chain-of-bridges contract:
   each new bridge enters a fresh consilium, and the original claim
   only upgrades when every bridge in the chain is ACCEPTED.
5. **Per-claim replay archive.** v1.3 returns one
   `replay_hash` per session. v1.4 should consider an archive of
   accepted-bridge hashes so the next mutation cycle can verify
   that the ledger entry it sees still matches the stored chain.
6. **Domain library versioning.** A change to
   `_METAPHOR_LIBRARY` invalidates every historic NEEDS_MORE_PREMISES
   verdict that depended on that pattern. v1.4 should stamp every
   `ConsiliumReplay` with a `library_version` so a re-replay surfaces
   the drift instead of silently agreeing.
7. **Jury awareness of CONSILIUM_ACCEPTED.** The v0.6+ DelphiJury
   still does not consume `CONSILIUM_ACCEPTED` events. v1.4 could
   let the Methodiker require a passing consilium on every bridge
   referenced by a Proposal.
8. **Bridge chains across runs.** Today a consilium is per-bridge.
   A claim that needs two bridges (e.g. P3 → bridge → another bridge)
   has no documented chain semantics. v1.4 needs a formal
   "BridgeChain" type whose replay_hash is the deterministic union
   of the per-bridge hashes.
9. **Auto-test obligations for new bridge libraries.** v1.3 ships
   pattern libraries as plain dicts. v1.4 should require that every
   new pattern carry a `test_id` so the audit log can verify the
   pattern's introduction was test-covered before it ever fired in
   production.
