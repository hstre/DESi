## Frage

> *"Is recursion decorative because it is useless —
>  or because recursive cases never reach the recursion gate?"*

v2.3 hat gezeigt, dass `depth_reached = 0` in 29/30 multi-step
Fällen. v2.4 lokalisiert, **wo** im Pipeline-Trichter die Fälle
sterben, bevor sie den `RecursiveResolver` überhaupt erreichen.

---

## Methodik

Zwei dünne Proxy-Wrapper, lokal in `bridge_audit/runner.py`:

* `_TracingAuditor(LogicalAuditor)` — speichert pro `audit()`-Aufruf
  Premise-Count, Premise-Kinds, Audit-State, Bridge-Count, Bridge-Kinds.
* `_TracingConsilium(BridgeConsilium)` — speichert pro
  `deliberate()`-Aufruf den `Verdict`-Wert und die `blocking_roles`.

Beide leiten alle Methodenaufrufe per `__getattr__` an die echten
Komponenten weiter. Semantik unverändert; `RecursiveResolver` sieht
die Wrapper, der Walk verläuft identisch zu v2.3.

`classify_loss_stage()` projiziert jede Trace auf **genau einen**
Wert der geschlossenen 9er-`LossStage`-Enumeration.

## Aufgabe 8 — Reflexion (nur aus Daten)

### Wo stirbt die Rekursion?

> **Sie stirbt am Parser, nicht am Resolver.**

23 von 30 Cases erreichen den `RecursiveResolver` **gar nicht erst**.
Der Auditor extrahiert Premises (1–4 pro Case), aber der Resolver
labelt das Resultat trotzdem mit `blocking_reason =
PARSER_UNSUPPORTED_FORM` — d.h. die Premise-Extraktion hat
strukturell Sätze gefunden, aber die *verbindende Inferenz-Form*
("Therefore X. Therefore Y.") wird als nicht-unterstützte Eingabe
gewertet. Die Rekursion ist hier nicht *unbrauchbar*, sondern
**unerreichbar**.

Die wenigen Cases, die weiterkommen:

* **R5_02–R5_06** (5/30): parser ok, Bridge erzeugt, Consilium
  akzeptiert ("accept_as_bridge"), Resolver wird betreten — kommt
  aber bei `depth_reached = 0` zurück und labelt `BLOCKED`.
  **Strukturelle Zykel-Erkennung feuert nicht**, weil keine
  Expansionsknoten existieren, an denen ein Zyklus überhaupt
  sichtbar werden könnte.
* **R4_06** (1/30): einzige `audit_reject_loss`. Der Auditor
  erkennt die Kontradiktion bei "All rectangles are squares" am
  Audit-Stage. Das ist die *einzige* Stelle, an der DESi eine
  bewusste epistemische Entscheidung trifft.
* **R4_04** (1/30): einzige `unknown_loss`. Premise-Count = 1,
  Bridge-Count = 0, aber Consilium wird aufgerufen, depth=1,
  Endzustand `COMPLETE`. Das ist die einzige Stelle, an der die
  Rekursion zündet — und sie zündet falsch. Denying the antecedent
  wird akzeptiert.

### Welche Schicht ist die nächste Intervention?

Datenbasiert klar **parser-side**:

| Datenpunkt | Quelle |
| --- | --- |
| `parser_loss = 23/30 (77 %)` | dominant loss stage |
| 6/6 R1-Cases (depth-1-Chain) → parser_loss | per-category |
| 6/6 R2-Cases (depth-2-Chain) → parser_loss | per-category |
| 6/6 R3-Cases (depth-3-Chain) → parser_loss | per-category |
| 4/6 R4-Cases (Kontradiktion) → parser_loss | per-category |
| Premise-Count > 0 in 30/30 Cases | runtime trace |

Die Auditor-Premise-Extraktion arbeitet — sie erkennt 1 bis 4
Premises pro Case. Aber die **Inferenz-Regel-Auswahl** im Audit,
die "Therefore"-Brücke zwischen zwei Premises zu einer
strukturierten Inferenz, schlägt fehl und propagiert nach oben als
`PARSER_UNSUPPORTED_FORM`. Konkret: keine `SYLLOGISM`/`IMPLICATION`/
`TRANSITIVITY`-Regel matcht auf "It is raining. Therefore the
street is wet."

> **Empfehlung v2.5**: nicht Resolver, nicht Consilium, nicht
> Bridge — sondern **Inferenz-Regel-Coverage-Diagnose**. Welche
> Regel-Form fehlt, damit eine simple `causal_chain`-Inferenz
> ("Therefore X. Therefore Y.") nicht mehr als
> `PARSER_UNSUPPORTED_FORM` blockt? Diese Frage ist read-only
> beantwortbar (Regel-Hit-Statistik über das v2.3-Set), bevor
> irgendein Patch vorgeschlagen wird.

Die `cycle_not_recognized`-Fälle sind eine *Folge* des gleichen
Defekts: ohne Inferenz-Regel-Match wird der Resolver bei depth 0
verlassen, und Zykel-Detektion ist strukturell unmöglich, wenn
kein Knoten expandiert wurde.
