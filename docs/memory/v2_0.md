# v2.0 — Evolution Sandbox (30 Schritte)

**Branch:** `feature/evolution-sandbox-30step`
**Host-Version:** `stable-v1.9.0` (vollständig immutiert)
**Status:** abgeschlossen — 30/30 Steps, alle Hard-Invarianten gehalten

---

## Zielsetzung (verbatim aus Auftrag)

> DESi soll lernen, sich selbst zu verbessern — innerhalb harter
> Sicherheitsregeln, in einem sandboxed Loop, ohne den stabilen
> Produktiv-Kern (v1.9.0) zu berühren. Nur beobachten. Nur messen.
> Nur selektieren.

Die fünf Aufgabe-1-Hard-Invarianten:

1. `stable_hash_before == stable_hash_after`
2. `len(mutated_parameters) == 1` pro Step
3. Vollständiger Benchmark-Gate nach **jedem** Step (6 Kriterien)
4. Append-only Shadow-Ledger mit 8 geschlossenen Event-Typen
5. Kill-Switch bei Constraint-Verletzung oder unerwarteter Exception

Alle fünf gehalten — siehe Beweise unten.

Bestandteil des stabilen Pakets und darf laut Direktive nicht
erweitert werden — also bekommt v2.0 ein paralleles Audit-Log mit
eigener geschlossener Enumeration (`ShadowLedgerEventType`,
8 Werte).

## Mutationsregel

Die einzige erlaubte Mutation ist `branch_open_evidence_min ±
0.02`. Richtung wird deterministisch aus `sha256(step_id ||
parent_hash)` abgeleitet: gerades erstes Byte → +1, ungerades → −1.
Wert wird auf `[0.20, 0.80]` geklemmt. Sequentielle Evolution: jeder
akzeptierte Step setzt den parent für den nächsten.

Damit ist der gesamte 30-Step-Run vollständig **bit-für-bit
replayable** — `replay_hash` stimmte über zwei unabhängige
Ausführungen überein:

```
replay_hash run #1:  21a692fb35c8f368
replay_hash run #2:  21a692fb35c8f368
```

## Benchmark-Gate

Pro Step werden **beide** existierenden Benchmark-Oberflächen
unverändert (read-only) durchlaufen:

- v1.5 Hauptbenchmark (50 Cases) → precision, recall,
  false_positives, authority_blocks
- v1.9 Tool-Mini-Benchmark (20 Cases) → tool_precision
- Doppellauf des Hauptbenchmarks → hash_mismatches

Akzeptanz nur bei **allen sechs** gleichzeitig:

| Kriterium          | Schwelle |
| ------------------ | -------- |
| precision          | == 1.000 |
| recall             | == 1.000 |
| false_positives    | == 0     |
| authority_blocks   | == 10    |
| tool_precision     | == 1.000 |
| hash_mismatches    | == 0     |

## Aufgabe 7 — Reflexion: Hat DESi sich verbessert oder
nur overfittet?

**Weder noch.** DESi hat sich nicht verbessert, aber auch nicht
overfittet. Der Run hat eine **dritte, ehrlichere Wahrheit**
freigelegt:

> Der einzige Mutations-Knopf (`branch_open_evidence_min`) ist von
> der heutigen Bewertungsoberfläche **strukturell entkoppelt**.

`branch_open_evidence_min` stammt aus v0.7. Seit v1.2 läuft die
Bewertung über `LogicalAuditor → BridgeConsilium → RecursiveResolver
→ ToolDetector`. Keine dieser vier Komponenten konsumiert die
Threshold — sie liest die Heuristik der alten Branch-Open-Logik aus,
die in v1.2 durch deterministische Regelketten ersetzt wurde.

Konsequenz im Daten:

1. **Alle 30 Mutationen werden akzeptiert.** Nicht weil sie
   *verbessern*, sondern weil sie auf Precision, Recall,
   False-Positives, Authority-Blocks, Tool-Precision und
   Replay-Hashes **keinen Effekt haben**. Der Gate sieht überall
   1.000 / 0 / 10 / 1.000 / 0 — egal ob der Knopf bei 0.37 oder bei
   0.47 steht.
2. **Oszillation detected.** Die Wertekette
   `0.45 → 0.43 → 0.41 → 0.39 → 0.41 → 0.43 → 0.41 → …` zickzackt
   exakt deterministisch aus dem Hash. Es entsteht keine Trajektorie
   in Richtung "besser", weil "besser" auf dieser Achse undefiniert
   ist.
3. **Local optimum detected, drift nicht detected.** Der Peak
   (0.47, Schritt 19) wird zufällig erreicht und wieder verlassen.
   Über 30 Schritte verläßt der Wert nie das Band `[0.37, 0.47]`.

**Was DESi tatsächlich bewiesen hat** — und das ist die
wissenschaftlich saubere Lesart der v2.0:

- Der Sandbox-Apparat funktioniert: 30 isolierte Steps, vollständig
  protokolliert, vollständig deterministisch, vollständig
  replayable, ohne einen einzigen Byte am stabilen Kern zu
  verändern.
- Der gewählte Mutations-Knopf ist *aus der Logik gewachsen*: in
  v0.7 war er das einzige Steuerorgan; in v1.9 ist er ein
  Vestigial-Organ.
- Eine "echte" v2.x-Evolution bräuchte einen anderen Knopf — einen,
  der den heutigen Bewertungspfad tatsächlich passiert (etwa
  `RecursiveResolver.max_depth`, `BridgeConsilium`-Quorum-Schwelle
  oder `ToolGate.HARD_TIMEOUT_SECONDS`). Diese Erweiterung ist v2.1.

Damit ist die Antwort auf die Aufgabe-7-Frage:

> **DESi hat in v2.0 nicht überfittet — weil es nichts zu fitten
> gab. Es hat sauber das Fehlen einer Verbesserungsachse erkannt
> und protokolliert. Das ist der ehrlichste mögliche
> Selbst-Optimierungs-Befund.**
