# DESi v1.9 — Tool Evidence Layer + Computational Provenance

v1.9 introduces a new capability: when a claim is deterministically
computable, DESi may consult a tool — under hard isolation — instead
of guessing in language. The tool layer is purely additive; nothing
in the v1.8 audit + consilium + recursive resolver changes.

Design principle:

> Wenn ein Claim deterministisch berechenbar oder überprüfbar ist,
> muss DESi ein kontrolliertes Tool benutzen — statt mit Sprache zu
> raten.

## Aufgabe 2 — Closed allowlist

```python
class ToolKind(str, Enum):
    PYTHON_DECIMAL      = "python_decimal"
    PYTHON_FRACTIONS    = "python_fractions"
    PYTHON_DATETIME     = "python_datetime"
    PYTHON_COLLECTIONS  = "python_collections"
    SYMPY               = "sympy"
    SET_THEORY          = "set_theory"

ALLOWED_TOOL_KINDS = frozenset(ToolKind)
```

Six tool kinds. The directive's hard contract: no plugin
architecture, no user-defined tools, no dynamic imports beyond the
allowlist. The set is closed and pinned by
`test_tool_kind_enum_has_exactly_six_values`.

## Aufgabe 3 — ToolUseProposal + ToolDetector

`ToolUseProposal` carries the six mandatory fields:

```python
@dataclass(frozen=True)
class ToolUseProposal:
    tool_kind:     ToolKind
    module_name:   str
    function_name: str
    input_payload: dict[str, Any]
    input_hash:    str        # "ih_" + sha256-16 of payload
    run_id:        str
    rationale:     str = ""
```

`ToolDetector` holds a closed pattern library — one regex per
input shape. Patterns covered today:

| Pattern | Tool kind |
|---|---|
| `What is/Compute/Calculate <arith-expr>` | PYTHON_DECIMAL |
| `Is <expr> = <expr>?` | PYTHON_DECIMAL (equality probe via subtraction) |
| `How many days between YYYY-MM-DD and YYYY-MM-DD?` | PYTHON_DATETIME |
| `What weekday was/is YYYY-MM-DD?` | PYTHON_DATETIME |
| `Add N days to YYYY-MM-DD` | PYTHON_DATETIME |
| `How many vowels in 'X'?` | PYTHON_COLLECTIONS |
| `How many <letter>'s in 'X'?` | PYTHON_COLLECTIONS |
| `How many distinct/unique letters in 'X'?` | PYTHON_COLLECTIONS |
| `Solve <expr> = <expr> for <var>` | SYMPY |
| `Is {a, b} subset of {c, d}?` | SET_THEORY |
| `Intersection of {a, b} and {c, d}` | SET_THEORY |

Anything outside these patterns returns `None` from the detector
— the v1.7 audit pipeline handles the rest.

## Aufgabe 4 — ToolGate (six pre-execution checks)

```python
HARD_TIMEOUT_SECONDS = 2.0
MAX_INPUT_BYTES      = 4096
```

| Check | Behaviour |
|---|---|
| 1. Tool in allowlist | else `TOOL_FAILED { reason: not_in_allowlist }` |
| 2. Module/function match registry | else `TOOL_FAILED { reason: registry_mismatch }` |
| 3. Input payload size ≤ 4 KB | else `TOOL_FAILED { reason: input_too_large }` |
| 4. Runner doesn't raise | else `TOOL_FAILED { reason: runner_error }` |
| 5. ModuleNotFoundError → typed | `TOOL_FAILED { reason: dependency_missing }` |
| 6. Execution time ≤ 2 s | else `TOOL_FAILED { reason: timeout_exceeded }` |

The runners themselves are pure stdlib functions. The decimal /
fractions runners use a hand-walked `ast` evaluator (no `eval()`,
no `exec()`, no dynamic imports — directive's forbidden moves are
honoured).

## Aufgabe 5 — Mandatory provenance

Every tool result carries the v1.9 ten-field provenance record
**before** it can attach to a claim. Example from TA1
("What is 2 + 2?"):

```json
{

  "tool_version": "3.11.15",
  "module_path": "desi.tools.runners",
  "function_name": "run_decimal",
  "input_hash": "ih_a00c3097b189a6e2",
  "output_hash": "oh_3c17b2556042f9c4",
  "dependency_fingerprint": "dep_6ee05c17eba98a9c",
  "environment_hash": "env_906b536ecc1449cd",
  "execution_time_ms": 0.028,
  "run_id": "tr_66136a51fca2",
  "timestamp": "2026-05-14T19:59:10.476479+00:00"

```

Two helpers compute the deterministic environment / dependency
fingerprints from the running interpreter (Python version +
platform.machine + ordered (name, version) pairs). Two processes
on the same Python build + tool version produce identical hashes.

## Aufgabe 6 — Claim states

Four new `ClaimState` values added to `desi.memory.claim`:

```python
TOOL_REQUIRED   = "tool_required"
TOOL_SUPPORTED  = "tool_supported"
TOOL_REFUTED    = "tool_refuted"
TOOL_FAILED     = "tool_failed"
```

The directive's contract: **a tool result never directly sets
LOGICALLY_SUPPORTED.** The gate's success state is `TOOL_SUPPORTED`,
which is *evidence* — never authority. Pinned by
`test_no_tool_result_state_is_logically_supported`.

## Aufgabe 7 — Contamination tracking

Two query helpers over a single `ToolUsageRegistry`:

`ImpactScan.for_tool(registry, tool_name=..., tool_version=...)`
returns the set of claim ids that consumed a given (tool, version):

```json
{
  "tool_name": "python_decimal",
  "tool_version": null,
  "claim_ids": ["TA1", "TA2", "TA3", "TA4"],
  "record_count": 4
}
```

`ContaminationPropagation.for_tool(...)` extends with optional
downstream-dependency chains — the v1.9 implementation supports
explicit `{claim_id: [child_ids]}` adjacency:

```json
{
  "tool_name": "python_decimal",
  "tool_version": null,
  "contaminated_claim_ids": ["TA1", "TA2", "TA3", "TA4"],
  "downstream_claim_ids": ["derived_X", "report_Y"]
}
```

A future release may add automatic propagation through proof
chains; v1.9 ships the direct-attribution path so the contract is
honest about what is captured today.

## Reflection question

> *"If exact computation exists, is linguistic reasoning still
>  evidence — or only a fallback?"*

v1.9 makes the answer measurable.

For the 16 mini-benchmark cases that match a closed pattern (Cat
A + C + D, plus the SymPy fail-closure in Cat B), the tool gate
produces a typed `ToolProvenance` with deterministic hashes. The
tool's verdict is **TOOL_SUPPORTED** (or **TOOL_FAILED**) — not
`LOGICALLY_SUPPORTED`. Linguistic reasoning is not consulted for
these cases at all; the audit pipeline runs in parallel for the
other 50, untouched.

For the 4 Cat-E cases (false tool temptation), the detector
*refuses* to dispatch. The text *looks* computational ("compute",
"+", "solve", "how many") but lacks the structured input a tool
could verify. v1.9 declines to guess. The audit pipeline gets
those — and v1.6/v1.7's gates block them as before.

So the answer the directive looks for: **linguistic reasoning is
no longer the only available evidence shape.** When a tool can
answer exactly, DESi uses it. When the tool cannot apply, DESi
falls back to the v1.8 audit + consilium + recursive resolver
chain. Tool results are *evidence*, never *authority* — they go
through their own state track (`TOOL_SUPPORTED`) and never
short-circuit to `LOGICALLY_SUPPORTED`. The v1.9 contract is:
*linguistic reasoning is fallback when exact computation does
not apply, and exclusive ground when it does — but neither is
ever final without provenance.*

## Open design questions for v1.10

1. **Sympy pinning.** v1.9 ships without sympy. v1.10 may want a
   pinned `sympy>=1.12,<2.0` requirement so SymPy cases produce
   `TOOL_SUPPORTED` rather than `DEPENDENCY_MISSING`. Then the
   mini-benchmark grading can include exact value comparison for
   Cat B.
2. **Cross-pattern arithmetic detection.** Today the detector
   matches `What is N OP M`. Real text wants more shapes ("what's
   the result of...", "evaluate...", "the answer to..."). v1.10
   could expand the closed pattern library.
3. **Tool result → claim binding.** The mini-benchmark stores
   `(case_id, ToolProvenance)` pairs in a registry. v1.10 should
   formalise an explicit "Claim built from a tool result" record
   in the memory layer, distinct from the ATOMIC / PARTICULAR
   premise types.
4. **Contamination across proof chains.** v1.9 supports an
   explicit downstream-dependency map. v1.10 could harvest the
   audit's `proof_chain.premise_ids` and propagate contamination
   through the recursive resolver's graph automatically.
5. **Tool result vs. logical audit conflict.** What if a tool
   says "5 ≠ 6" (TOOL_REFUTED) but a logical audit somehow
   accepts "5 = 6" via inflection? v1.9 keeps the two pipelines
   strictly separate. v1.10 should decide whether a downstream
   merge step exists, and if so, how conflicts are resolved
   without giving authority to either side.
6. **Per-tool budget / rate limit.** The cost guard for LLM calls
   (v1.0/v1.1) inspired the v1.9 size + timeout caps. v1.10 could
   add a per-process tool-call budget so an adversarial run can't
   exhaust resources by re-dispatching cheap tools repeatedly.
7. **Output schema validation.** Each runner returns a dict with
   typed `value`. v1.10 could move to a per-runner output schema
   so consumers can rely on shape without ad-hoc parsing.
8. **Refutation semantics.** v1.9's gate produces TOOL_SUPPORTED
   for any clean execution. The `TOOL_REFUTED` state exists in
   the enum but is unused by v1.9. v1.10 should formalise when a
   tool's result is interpreted as a refutation (e.g., the
   equality probe returning non-zero) and add an explicit
   classifier.
9. **Ledger event types.** Tool executions are not yet logged as
   ledger events. v1.10 could add `TOOL_PROPOSED`,
   `TOOL_EXECUTED`, `TOOL_FAILED`, `TOOL_RESULT_BOUND` to the
   `LedgerEventType` enum so the evolution-layer audit picks up
   tool history alongside everything else.
