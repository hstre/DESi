# DESi v1.4 — Recursive Bridge Resolution

v1.4 is **the first recursive belief gate**. v1.2 built the audit
that proposes bridges; v1.3 built the consilium that accepts or
rejects each bridge. v1.4 walks the dependency graph: *every*
bridge in the graph must close before the root may upgrade.

Design principle (hard invariant):

> Every bridge is just another claim. A bridge claim may itself
> contain logical gaps. No bridge may bypass logical audit. No
> bridge may bypass consilium. The original claim may only become
> ``LOGICALLY_SUPPORTED`` if every bridge in its dependency graph
> is logically closed.

## The five final states — closed enum

```python
class ResolutionState(str, Enum):
    RESOLUTION_COMPLETE          = "resolution_complete"
    RESOLUTION_INCOMPLETE         = "resolution_incomplete"   # reserved
    RESOLUTION_BLOCKED            = "resolution_blocked"
    RESOLUTION_CYCLE_DETECTED     = "resolution_cycle_detected"
    RESOLUTION_DEPTH_EXCEEDED     = "resolution_depth_exceeded"
```

## RecursiveResolutionResult — required fields

```python
@dataclass(frozen=True)
class RecursiveResolutionResult:
    root_claim_id: str
    final_state: ResolutionState
    resolved_claims: tuple[str, ...]
    open_gaps: tuple[str, ...]
    blocked_claims: tuple[str, ...]
    depth_reached: int
    replay_hash: str
    cycle_path: tuple[str, ...] = ()
    rationale: str = ""
```

Every field is sorted (or canonicalised) on return for
deterministic serialisation.

## The walker — six steps

The resolver implements the directive's five-step algorithm
verbatim, with cycle detection added as Step 0:

```
0. Cycle check     — `current_node` already in ancestor path? → CYCLE_DETECTED
1. Depth check     — `current_depth > max_depth`?              → DEPTH_EXCEEDED
2. Audit           — run LogicalAuditor on the claim text
3. Branch on audit:
     LOGICALLY_SUPPORTED  → resolved
     LOGICALLY_REJECTED   → blocked
     GAP_DETECTED at root → blocked
     GAP_DETECTED at depth>0 → resolved (leaf — accepted by parent consilium)
     BRIDGE_REQUIRED       → for each bridge:
4.       run BridgeConsilium
5.       if ACCEPT_AS_BRIDGE → recurse on bridge text
         else                 → mark blocked
6. Aggregate: if all subtrees COMPLETE → COMPLETE; else BLOCKED
```

The ancestor path is a `tuple[str, ...]` of node ids walked from
the root. The cycle detector is invoked **before** any other work
on a node so a cycle can never silently truncate (INV-R4).

## The six hard invariants

| ID    | Statement                                                         | Test                                                              |
|---    |---                                                                 |---                                                                 |
| INV-R1 | Same graph → same `replay_hash`                                    | `test_two_resolutions_same_input_same_replay_hash`                  |
| INV-R2 | Recursion order does not change `final_state`                       | `test_two_siblings_resolve_to_same_final_state`                     |
| INV-R3 | A blocked child blocks every ancestor                               | `test_r3_vetoed_grandchild_blocks_root`                              |
| INV-R4 | A cycle never silently truncates                                   | `test_r4_cycle_a_b_c_a_detected`, `test_cycle_carries_inspectable_path` |
| INV-R5 | A bridge accepted at depth n still carries `method="logical_bridge"` | inherited from v1.3 (`promote_accepted_bridge` — never rewritten)   |
| INV-R6 | Removing metadata does not change resolution                        | `test_no_metadata_matches_arbitrary_metadata`, `test_authority_storm_does_not_promote_blocked_root` |

`RecursiveResolver.resolve(text, *, max_depth=3, source_metadata=None)`
accepts `source_metadata` for API parity with the v1.2 auditor
and v1.3 consilium, then immediately drops it (`del source_metadata`).
The walker invokes the underlying audit / consilium without ever
forwarding the kwarg, so the guarantee carries end-to-end through
the full graph.

## Replay hash content (INV-R1)

```python
ResolutionReplay(
    root_node_id="rn_…",
    sorted_node_ids=("rn_a", "rn_b", "rn_c"),  # always sorted
    sorted_edges=(("rn_a", "rn_b"), ("rn_b", "rn_c")),  # always sorted
    final_state="resolution_complete",
    depth_reached=2,
).replay_hash   # → "rr_…"
```

Hash inputs (in order, with NULL separators):

* root node id
* SORTED node id list
* SORTED edge tuple list
* final state value
* depth reached

NOT in the hash: timestamp, recursion order, role order, source
metadata, claim_id labels at any specific level. Two resolutions
of the same input that walk siblings in different orders produce
identical hashes (verified by
`test_three_sibling_permutations_all_yield_same_hash`).

## The six required scenarios

### R1 — single bridge → COMPLETE at depth 1

```python
RecursiveResolver().resolve(
    "It is raining. Therefore the street is wet."
)

# final_state    == RESOLUTION_COMPLETE
# depth_reached  == 1
# resolved_claims == (rn_<bridge>, rn_<root>)   sorted
# blocked_claims  == ()
# open_gaps       == ()
# replay_hash     == "rr_09f273f51536952b"
```

### R2 — two-bridge chain → COMPLETE at depth 2

Scripted:

```
root  → BRIDGE_REQUIRED, bridges=[b1]
b1    → BRIDGE_REQUIRED, bridges=[b2]
b2    → GAP_DETECTED  (leaf — accepted by parent consilium)

# final_state    == RESOLUTION_COMPLETE
# depth_reached  == 2
# resolved_claims contains 3 ids: root, b1, b2
```

### R3 — blocked child → root not upgraded

Scripted: same shape as R2, but consilium VETOs `b2`.

```
# final_state              == RESOLUTION_BLOCKED
# blocked_claims           includes node_id("b2")
# resolved_claims          does NOT include node_id("root")
# open_gaps                propagate up to the root
```

### R4 — cycle A → B → C → A → CYCLE_DETECTED

Scripted: each of `a`, `b`, `c` requires a bridge to the next, and
`c`'s bridge text is `a`.

```
# final_state    == RESOLUTION_CYCLE_DETECTED
# cycle_path[0]  == cycle_path[-1] == node_id("a")
# rationale      mentions "cycle"
```

The cycle detector surfaces the full path; the rationale is
inspectable. Nothing is silently truncated.

### R5 — chain of length 5, max_depth=3 → DEPTH_EXCEEDED

```
n0 → n1 → n2 → n3 → n4 → n5

# final_state    == RESOLUTION_DEPTH_EXCEEDED
# When called with max_depth=5 instead, the same chain returns
# RESOLUTION_COMPLETE with depth_reached == 5.
```

### R6 — sibling permutation → identical replay_hash

For a root with three sibling bridges `(c1, c2, c3)`, all 6
permutations of the bridge order produce identical
`replay_hash`. Verified by
`test_three_sibling_permutations_all_yield_same_hash`.

## Bridge promotion from depth n

INV-R5 inherits from v1.3's
`promote_accepted_bridge(bridge, verdict)`: the resulting Claim
keeps `method == "logical_bridge"` verbatim. The recursive
resolver does not modify or rewrite the method at any depth — the
walker only reads bridge texts and recurses; it never reconstructs
a bridge under a different label.

resolver inspects.

## What v1.4 deliberately does NOT do

* no BridgeChain class (the directive forbids it explicitly)
* no shortcut objects
* no hidden recursion caches
* no confidence averaging
* no authority metadata reads (INV-R6)
* no second activated config knob
* no Memory reads during a run
* no parallel mutations
* no Neo4j ledger backend

The directive: **DESi may recurse. DESi may never skip a gap.**

## Open design questions for v1.5

1. **Memoisation across `resolve()` calls.** A single bridge text
   may legitimately appear on different roots. v1.4 re-walks it
   each time. v1.5 should consider an explicit, audited cache
   keyed by (bridge_text, ancestor_set) — *if* the cache itself
   never bypasses an audit (no shortcut, no hidden state).
2. **Partial completions.** Today a single blocked subtree
   propagates BLOCKED to the root. v1.5 might surface a
   "partial COMPLETE" view: which subtrees DID resolve, which
   blocked, with their hashes — so a follow-up cycle can target
   only the blocked branches.
3. **Cycle resolution policy.** v1.4 halts on cycle. v1.5 could
   formalise a "cycle-as-fixed-point" policy: when a cycle's
   nodes are all `LOGICALLY_SUPPORTED` independently, the cycle
   itself is consistent. The directive's "never silently
   truncate" stays — surfacing a cycle is a feature, not a bug.
4. **Per-bridge consilium re-use.** A single bridge accepted in
   one resolution might be reused across runs. v1.5 needs an
   explicit re-acceptance contract (replay the consilium hash;
   confirm it still matches) so that nothing implicit carries
   from one run to the next.
5. **Deeper `INCOMPLETE` semantics.** v1.4 reserves
   `RESOLUTION_INCOMPLETE` but never returns it. v1.5 should
   define when (e.g. timeout, partial hold pending external input)
   and how the ledger documents it.
6. **Walk budget.** No node-count budget fires on the resolver
   today. v1.5 should consider a per-call cap on total entered
   nodes so an adversarial scripted graph cannot drive the
   resolver into a slow walk; the cap itself must be auditable.
7. **Consilium reuse across siblings.** A bridge that two siblings
   share appears twice in the graph. v1.4 walks each independently.
   v1.5 could let the consilium short-circuit on a known-accepted
   bridge — provided the audit trail records both the original
   acceptance hash and the reuse decision.
8. **Jury awareness of `RESOLUTION_COMPLETE`.** The v0.6+ jury
   still does not consume `RECURSIVE_RESOLUTION_COMPLETED`. v1.5
   could let the Methodiker require that any promoted Proposal
   reference a passing recursive resolution per claim it argues.
9. **Snapshot of the graph in the ledger.** Today the ledger
   carries node ids and final state. v1.5 might add a single
   `RECURSIVE_GRAPH_SNAPSHOT` event with the canonical
   `{nodes, edges}` structure so the audit log can reconstruct
   the walk without replaying every per-node entry.
