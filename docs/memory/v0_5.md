# DESi v0.5 — Constitutional Delphi Evolution Layer

v0.5 introduces a governance-controlled self-improvement loop on top
of the v0.4 observation framework. DESi may now propose changes to
its own configuration after observing its performance, but every
proposed change must pass through a constitutional process before it
becomes stable.

Design principle:

> DESi darf sich verbessern. DESi darf sich nicht unkontrolliert
> verändern.

Every mutation is isolated, auditable, rollback-capable,
justification-required, and veto-capable.

## New surface

```
src/desi/evolution/
  __init__.py        public API
  reflection.py      ReflectionEngine + ReflectionReport + ReflectionFinding
  proposal.py        MutationProposal + MutationTarget
  sandbox.py         CloneSandbox + StableConfig + default_stable()
  evaluation.py      MutationEvaluation + report + 5 adversarial patterns
  jury.py            DelphiJury + JuryMember + JuryReview + Vote + Veto
  promotion.py       Snapshot + PromotionGate + promote() + rollback()
```

## The constitutional process

```
ReflectionEngine          → ReflectionReport
       │
       ▼
MutationProposal          (problem, hypothesis, target, config_delta,
       │                   expected_improvement, rollback_conditions,
       │                   motivating_findings)
       ▼
CloneSandbox.apply()      → clone with new knobs; stable untouched
       │
       ▼
MutationEvaluation.run()  → MutationEvaluationReport
       │                   (Pflicht S2/S6/S7,
       │                    Adversarial × 5,
       │                    Regression S1–S7)
       ▼
DelphiJury.deliberate()   → JuryDecision
       │                   (round 1 independent reviews,
       │                    round 2 final votes with optional Veto)
       ▼
PromotionGate             (no valid veto, all suites green, quorum 4/5)
       │
       ├─ pass → promote() → Snapshot + new StableConfig
       │
       └─ fail → PromotionError(reasons); stable unchanged
```

## ReflectionEngine

Six deterministic detectors:

| detector              | flags                                            |
|---                    |---                                                |
| unnecessary_branches  | > 4 unique focus_claim_ids opened in a single run |
| repeated_revisions    | > 2 revisions on the same claim_id               |
| dead_end_claims       | claims with zero in- or outgoing relations       |
| late_guard_triggers   | guard events in the final 25% of the timeline    |
| unstable_path         | high operator-switch rate                        |
| unresolved_conflicts  | odd CONTRADICTS count (missing mirror)           |

## MutationProposal

Required fields (pydantic, `extra="forbid"`):

```python
MutationProposal(
    parent_version="stable-v0.5.0",
    problem="Branches open too eagerly on weak evidence.",
    hypothesis="Raising branch_open_evidence_min from 0.30 to 0.45 "
               "should suppress speculative branches.",
    target=MutationTarget.BRANCH_HEURISTICS,
    config_delta={"guard_thresholds.branch_open_evidence_min": 0.45},
    expected_improvement="At least 30% fewer branch_opened events on "
                         "S5 under seed=42.",
    rollback_conditions=("revert if branch_opened on S2 falls below 1",),
    motivating_findings=("rf_branch_overopen",),
)
```

## DelphiJury — deterministic role personas, no LLM

Five roles, two rounds:

| role          | round-1 focus                                   |
|---            |---                                               |
| Replikator    | reproducibility, regression-suite stability      |
| Skeptiker     | rollback declared? config_delta non-empty?       |
| Methodiker    | motivating_findings cited?                       |
| Adversarial   | adversarial suite green?                          |
| Integrator    | promotion preconditions globally green?           |

Round 2: every member sees all round-1 reviews and casts a final vote
(`APPROVE` / `REVISE` / `VETO`). v0.5 directive forbids LLM-API and
internet access; this design respects that.

## Veto protocol

A veto is valid only if it carries `affected_claim`, `suspected_risk`,
`failure_case`, and `proposed_test`. Vetos missing any of these fields
are discarded by `JuryDecision`.

## Promotion preconditions

`PromotionGate.check()` enforces seven conditions: no valid vetos
outstanding; regression / adversarial / pflicht suites green;
performance_delta and path_quality_delta not negative; jury quorum
reached (≥ 4/5 APPROVE).

## What v0.5 deliberately does NOT do

* no real LLM API calls
* no internet access
* no self-modification of stable memory inside a clone
* no overwriting of stable claims
* no skipping the jury, no skipping the regression suite
* no silent promotions
