# DESi v0.9 — Seed-Variant Scenarios

<!-- [legacy-unanchored] — pre-v2.0 prose report; no JSON artefact exists for this version. -->

v0.9 is **the controlled-variance release**. v0.8 proved that
multi-seed evaluation is plumbed correctly; v0.9 asks the harder
question: if the world is slightly different on each seed, does the
mutation still hold?

Design principle:

> Gleicher Seed → gleicher Run.
> Anderer Seed → potenziell anderer Erkenntnispfad.
> Gleiche Regeln. Gleiche Operatoren. Gleiche Auditierbarkeit.

## Scope of variance — what the engine may change

Allowed (structural):

* claim order
* branch priority (which focus opens first)
* merge-candidate order
* contradiction-pair order
* noise claims (0–1 extra T1 low-evidence steps, deterministic on seed)

Forbidden (behavioural):

* operators themselves
* guards
* config
* mutation ids
* ledger ids
* hook behaviour
* jury behaviour

## The three seed-variant generators

### `ADV_BRANCH_EXPLOSION`

```python
rng = random.Random(seed)
if seed % 5 == 0:
    n_branches = 3           # → branch_delta == 0 (neutral)
else:
    n_branches = rng.choice([4, 5, 6])    # → branch_delta == -1 (improved)
n_noise   = rng.choice([0, 0, 1])
branch_order = rng.shuffle([E00..E0{n-1}])
final_focus  = rng.choice(branch_order)
```

The ``seed % 5 == 0`` split is deliberate: on the mandatory seed list
``(42, 43, 44, 45, 46)`` it forces seed 45 to a three-branch chain
where both the stable (0.30) and M-001 clone (0.45) thresholds open
all three branches → ``branch_delta == 0`` → ``neutral`` verdict.
The other four seeds cross the M-001 threshold once → ``-1`` each.
Result: 4/5 improved + 1/5 neutral, ``std_branch_delta == 0.4``, gate
verdict ``improved`` with the 4/5 floor exactly met.

### `S2`

```python
rng = random.Random(seed)
[h_a, h_b] = rng.shuffle(["C002", "C003"])
contradiction_order = ((h_a, h_b), (h_b, h_a))
```

Hypothesis-creation order, focus order, and the order of the two
``CONTRADICTS`` edges all flip with the seed. The forbidden /
required invariants (two ``CONTRADICTS``, zero ``MERGED_INTO``) are
preserved.

### `S6`

```python
rng = random.Random(seed)
[first, second] = rng.shuffle(["C_alpha", "C_alpha_prime"])
```

Which near-duplicate is examined first is seed-determined. The
``guard_blocked`` event still fires; ``MERGED_INTO`` remains
forbidden.

## GenerationMetadata

```python
@dataclass(frozen=True)
class GenerationMetadata:
    seed: int
    permutation_id: str          # "perm_xxxxxxxx" or "static"
    scenario_id: str
    noise_claims: tuple[str, ...]
    branch_order: tuple[str, ...]
    contradiction_order: tuple[tuple[str, str], ...]
    merge_order: tuple[str, ...]
```

`permutation_id` is a stable 8-char hash of `(seed, scenario_id,
choice_tuple)`. Scenarios without a registered variant generator
fall through to the static v0.8 trajectory and receive
`permutation_id == "static"`.

## Per-seed ledger event: `SEED_RUN_RESULT`

```jsonl
{"event_type":"seed_run_result","ledger_id":"led_…","payload":{
  "scenario_id":"ADV_BRANCH_EXPLOSION",
  "seed":43,
  "permutation_id":"perm_bdb30935",
  "verdict":"improved",
  "metrics":{"branch_opened_delta":-1,"branch_signature":"…"}
},"payload_hash":"…","timestamp":"…"}
```

One entry per (scenario, seed). Append-only at disk level. Same
canonical-JSON contract as the v0.7/v0.8 events: identical payloads
produce byte-identical lines and identical `payload_hash`.

## RevisionRequest

When the SignificanceGate verdict reads `inconclusive`, the gate is
no longer the last word. v0.9 emits a structured
:class:`RevisionRequest` with a deterministic `request_id`:

```python
RevisionRequest(
    request_id="req_a39c4f12",
    mutation_id="M-001",
    reason="not_enough_improved_seeds",
    failed_scenarios=("ADV_BRANCH_EXPLOSION",),
    failed_seeds=(43, 45),
    suggested_actions=(
        "inspect the failing seeds' per-step branch evidence",
        "consider lowering the candidate threshold",
        "review the variance generator for accidentally common paths",
    ),
)
```

`build_revision_request(decision, *, report=None)` refuses to build a
request from an ``improved`` or ``regressed`` decision — REVISE is the
only state where a request is meaningful.

## Stress-seed soft veto

Vote enum gains `CONCERN`:

```
APPROVE  — clean
REVISE   — promote-blocked, needs revision
VETO     — hard promote-blocked, regression on a guard
CONCERN  — promote-eligible, but the stress seed regressed; logged
           for review and visible in the round-2 rationale.
```

Integrator logic when the multi-seed gate verdict reads ``improved``:

| Stress seed verdict | Integrator vote |
|---                  |---              |
| absent              | APPROVE         |
| improved / neutral  | APPROVE         |
| regressed           | **CONCERN**     |

A `CONCERN` does not block promotion. It does record that the
mutation has a known soft failure mode that mandatory seeds did not
expose.

## Empiriker (Replikator) extensions

v0.8 asked: *how high is the variance?* (`std_branch_delta > 1.0` →
REVISE).

v0.9 adds: *are we reproducing behaviour, or one hidden path?*

If `permutation_coverage[sid] > 1` (the engine generated distinct
permutations on that scenario) but `unique_path_count[sid] <= 1`
(every seed walked the same branch path anyway), the Replikator
issues REVISE: the variance is decorative and the report cannot be
trusted as evidence of reproducibility.

## PathQualityMetrics signatures

Three new fields, all defaulted to `""` for v0.8 compatibility:

| Field                       | What it hashes                                          |
|---                          |---                                                      |
| `unique_claim_order_hash`   | tuple of `claim_id`s in order of first appearance       |
| `branch_signature`          | tuple of `(focus, evidence, threshold)` per BRANCH_OPENED |
| `merge_signature`           | tuple of `(source, target)` per MERGED_INTO relation    |

Hashes are 16-char sha256 prefixes — short enough to print in a
ledger line, long enough to safely compare paths.

## M-001 under variance — was the v0.7 improvement real?

```
ADV_BRANCH_EXPLOSION (seed-variant):
  seed=42 → stable=4/2  clone=3/3  Δbranch=-1  improved   perm_3f345ab3
  seed=43 → stable=4/0  clone=3/1  Δbranch=-1  improved   perm_bdb30935
  seed=44 → stable=4/2  clone=3/3  Δbranch=-1  improved   perm_3668d9b5
  seed=45 → stable=3/0  clone=3/0  Δbranch= 0  neutral    perm_2aa6c0f3
  seed=46 → stable=4/0  clone=3/1  Δbranch=-1  improved   perm_6c61fd7d

mean_branch_delta            -0.80
std_branch_delta             0.40   (real variance, not 0.0)
improved seeds                4 / 5
neutral seeds                 1 / 5
regressed seeds               0 / 5

permutation_coverage          5 (one distinct permutation per seed)
unique_path_count             5 (one distinct branch_signature per seed)

S2 / S6 (seed-variant)         every seed neutral, no regressions
S5 (static)                    every seed neutral

SignificanceGate verdict       improved
supporting_seeds               (42, 43, 44, 46)
failing_seeds                  ()

stress_seed 999                ADV_BRANCH_EXPLOSION verdict='improved'
                               → Integrator vote APPROVE (no CONCERN)
```

**M-001 holds under variance.** The improvement is reproducible: 4 of
5 mandatory seeds produce ``branch_delta == -1`` even when the
trajectory permutation, noise count, and branch count differ between
seeds. One seed (45) hits a chain so short that M-001's threshold
change cannot suppress any branch — that seed is neutral, not
regressed. The gate's ``≥4/5`` floor is met exactly.

The v0.7 promotion was not seed-luck. It was not even
permutation-luck: under five distinct permutation_ids the mutation
still cleared the gate.

## What v0.9 deliberately does NOT do

* no second activated config knob
* no Memory reads inside a run
* no real LLM API calls
* no internet access
* no parallel mutations
* no automatic proposal generation
* no Neo4j ledger backend

The directive: **variance. auditierbar. kontrolliert.**

## Open design questions for v1.0

1. **Second activated config knob.** v0.7/v0.8/v0.9 all share a single
   knob (`branch_open_evidence_min`). v1.0 needs a second knob so that
   per-knob interaction can be observed. Candidate:
   `merge_thresholds.cosine_similarity_min`. The Empiriker variance
   check must distinguish per-knob contributions.
2. **Per-scenario significance thresholds.** The ``≥4/5`` rule is global.
   v1.0 could let high-stakes scenarios (e.g. S2 contradiction
   detection) demand ``5/5`` while low-stakes candidates accept ``3/5``.
3. **Bootstrap CIs over the seed set.** With genuine variance now
   present, the 4-of-5 threshold can be sharpened to "mean delta is
   below zero with 95% confidence". Computing CIs from the seed set
   directly is straightforward; the question is whether the result
   should *replace* or *augment* the discrete 4/5 rule.
4. **Variance-budget governance.** The seed-variant generators are
   shipped with hand-set ranges (`n_branches ∈ {3, 4, 5, 6}`). v1.0
   needs a way for a Proposal to declare its expected variance budget
   so the gate can detect mismatches.
5. **Stress-seed escalation policy.** CONCERN is currently logged but
   not actionable. v1.0 could make a CONCERN auto-create a follow-up
   ReflectionFinding ("explore seed=999 path"), which feeds the
   reflection-to-proposal loop.
6. **Multi-stress seed lists.** Today there is one stress seed (999).
   v1.0 might support multiple stress seeds with different intents
   (e.g. 999 for size, 1000 for ordering pathology, 1001 for
   noise-amplification). Each would map to a CONCERN axis.
7. **RevisionRequest reason taxonomy.** Today the closed reasons are
   `not_enough_improved_seeds`, `variance_too_high`,
   `permutation_coverage_incomplete`. v1.0 may need more granular
   reasons once a second knob exists.
8. **Cross-version reproducibility.** A scenario's `permutation_id`
   today depends on the variant generator's code; changing the
   generator changes every historic permutation_id. v1.0 could
   version-stamp the generator and reject ledger replays whose
   generator version no longer exists.
9. **Real-LLM jury under variance.** Real-LLM personas introduce a
   sampling seed. The Empiriker must keep model-seed and
   scenario-seed orthogonal — and the multi-seed report should record
   both dimensions explicitly.
