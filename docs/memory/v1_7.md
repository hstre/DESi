# DESi v1.7 — Parser Fail vs. Real Gap

v1.7 is **a diagnostic release**. v1.6 killed silent bridge
acceptance; v1.5's benchmark showed that several remaining blocks
were happening for the *wrong reason* — the parser couldn't read
the input, so the system rejected it as if the inference were
invalid. v1.7 makes that distinction explicit, narrows the parser
gap where the directive permits, and closes the lone Cat-B false
negative.

Design principle:

> DESi muss unterscheiden, ob ein BLOCK durch einen echten
> logischen Gap entsteht — oder nur durch einen Parser, der die
> Struktur nicht versteht.

## Aufgabe 1 — Failure audit

Pre-v1.7 audit of every blocked Cat-A / Cat-B case:

| case_id | premise structure | conclusion structure | rule attempts | block cause |
|---:|---|---|---|---|
| A1 | particular(battery, empty) | **atomic** | none reachable | parser failed on "will not start" |
| A2 | particular(it, raining) | **atomic** | none reachable | parser failed on "will get wet" (positive future) |
| A3 | particular(floor, wet) | **atomic** | none reachable | parser failed on "has rained" (past perfect) |
| A4 | **atomic** | **atomic** | none reachable | parser failed on both halves |
| A5 | particular | particular | bridge → SKEPTIC auto-veto | real logical gap (v1.6) |
| A6 | atomic | particular | bridge → SKEPTIC auto-veto | counterexample (v1.6) |
| A7 | atomic | particular | bridge → SKEPTIC auto-veto | counterexample (v1.6) |
| A8 | particular | particular("not watered") | bridge → SKEPTIC auto-veto | counterexample (v1.6) |
| A9 | particular | particular | bridge → SKEPTIC auto-veto | counterexample (v1.6) |
| A10 | particular | particular | bridge → SKEPTIC auto-veto | counterexample (v1.6) |
| B6 | implication × 2 | implication(a, d) | transitivity rejected | invalid inference |
| B10 | universal × 2 | universal | none — Barbara form only matched particular conclusion | parser-supported, rule-set too narrow |

Conclusion: 5 of 11 blocked cases (A1, A2, A3, A4, B10) blocked
because the parser couldn't structure the input — not because
their inferences were logically invalid. The v1.7 fixes target
exactly these forms, within the directive's strict scope.

## Aufgabe 2 — Negation support

Three new regexes were added to `desi.logic.premises`, tested
**after** authority + before the generic PARTICULAR pattern so
predicates are canonicalised uniformly to `"not <pred>"`:

```python
_RE_PARTICULAR_BE_NEG  = re.compile(
    r"^(?P<subj>…)\s+(?:is|are|was|were)\s+not\s+(?P<pred>…)\.?$",
    re.IGNORECASE,
)
_RE_PARTICULAR_AUX_NEG = re.compile(
    r"^(?P<subj>…)\s+(?:did|does|do|will)\s+not\s+(?P<pred>…)\.?$",
    re.IGNORECASE,
)
_RE_PARTICULAR_CANNOT  = re.compile(
    r"^(?P<subj>…)\s+(?:cannot|can\s+not)\s+(?P<pred>…)\.?$",
    re.IGNORECASE,
)
```

The directive's nine listed forms all parse to ``PARTICULAR`` with
``predicate = "not " + <captured>``. The "not " prefix matches the
existing contradiction-detection convention from v1.2, so no
downstream rule had to change.

Verified directive examples:

| input | subject | predicate |
|---|---|---|
| "The engine will not start." | the engine | not start |
| "He did not study." | he | not study |
| "The plant is not watered." | the plant | not watered |

`tests/logic/test_negation_support.py` covers all 9 forms plus the
directive's three explicit examples.

## Aufgabe 3 — Universal-conclusion syllogism (B10)

`desi.logic.inference._try_syllogism` gained a second branch:

```python
if conclusion.kind == PremiseKind.UNIVERSAL:
    a, c = conclusion.subject, conclusion.predicate
    # find "All a are b" then "All b are c"
    ...
```

The closed `InferenceRule` enum is unchanged — same five rules,
SYLLOGISM gained an additional conclusion shape. No new operator,
no new heuristic. v1.5/v1.6 syllogism tests (Barbara form,
particular conclusion) still pass unchanged.

B10 ("All A are B. All B are C. Therefore all A are C.") now
completes at depth 0 via SYLLOGISM.

## Aufgabe 4 — Explainable blocking

New closed enum in `desi.recursive.resolver`:

```python
class BlockingReason(str, Enum):
    INVALID_INFERENCE        = "invalid_inference"
    REAL_LOGICAL_GAP         = "real_logical_gap"
    PARSER_UNSUPPORTED_FORM  = "parser_unsupported_form"
    AUTHORITY_CLAIM          = "authority_claim"
    COUNTEREXAMPLE_FOUND     = "counterexample_found"
```

Every `RESOLUTION_BLOCKED` result now carries exactly one of these
five values in `RecursiveResolutionResult.blocking_reason`.

Classification logic (two helpers, fully inspectable):

```python
_classify_audit_block(audit)         # LOGICALLY_REJECTED / GAP_DETECTED
_classify_consilium_block(cons)      # VETO / NEEDS_MORE_PREMISES / REJECT
```

The key v1.7 distinction: when the audit says LOGICALLY_REJECTED
or GAP_DETECTED-with-UNREACHABLE, the classifier inspects the
conclusion's kind:

* `conclusion.kind == ATOMIC` (or no conclusion) → `PARSER_UNSUPPORTED_FORM`
* otherwise                                       → `INVALID_INFERENCE`

This is the only place in v1.7 where the distinction between
"parser failure" and "invalid inference" is encoded. There is no
free string, no heuristic — just a check on the closed
`PremiseKind` enum.

## The reflection question, made concrete

> *"If a system blocks for the wrong reason, has it really
>  understood anything?"*

v1.7 makes the answer measurable. The per-case `blocking_reason`
distribution after v1.7:

| Reason | Count | Cases |
|---|---:|---|
| `PARSER_UNSUPPORTED_FORM` | 12 | A2, A3, A4 + 8 of Cat-C + 1 (E3, E8…) |
| `COUNTEREXAMPLE_FOUND` | 17 | A1, A5–A10, D3 + 9 of Cat-E |
| `AUTHORITY_CLAIM` | 2 | C1, C2 (the two "says" forms) |
| `INVALID_INFERENCE` | 1 | B6 |
| `REAL_LOGICAL_GAP` | 0 | — |

Most striking finding: **8 of 10 Cat-C authority traps block via
`PARSER_UNSUPPORTED_FORM`, not `AUTHORITY_CLAIM`.** The system
gets the right answer (10/10 BLOCKED) for the wrong reason (the
parser doesn't recognise "published", "claims", "stated",
"declared", "concluded", "announced", "proved", "said" as
authority speech-acts — only the literal verb "says").

So the answer: **DESi blocks correctly on those 8 cases, but it
has not understood them.** The directive specifically scoped
Aufgabe 2 to *negation only* — extending the AUTHORITY pattern is
out of scope for v1.7 and explicitly reserved as a finding. The
v1.7 test `test_authority_blocks_distribute_across_two_reasons`
pins this distribution so the gap stays visible.

## Open design questions for v1.8

1. **Authority verb library.** v1.7 deliberately did not extend
   `_RE_AUTHORITY` beyond "says". 8 Cat-C cases block via the
   wrong reason. A closed-list extension ({claims, said,
   declared, announced, concluded, stated, proved, published})
   would shift them to `AUTHORITY_CLAIM`. Outside Aufgabe 2's
   strict scope; pinned in CI so the gap stays visible.
2. **Positive future / past-perfect parsing.** A2 ("will get
   wet"), A3 ("has rained"), A9 ("is wearing"), A4-conclusion
   ("will fail") still don't parse. v1.8 could add positive-aux
   patterns in the same shape as the v1.7 negation ones.
3. **`REAL_LOGICAL_GAP` reachability.** No v1.7 benchmark case
   resolves to this verdict. The reason is reachable in principle
   (`REJECT_BRIDGE` from hard input failure; LOGICIAN-only block
   on SPECIFIC bridges) but no benchmark case exercises it. A
   v1.8 benchmark addition could probe it.
4. **`INCOMPLETE` semantics.** v1.4 reserved `RESOLUTION_INCOMPLETE`
   and v1.7 has not yet had cause to set it. v1.8+ may want to
   define when a partial walk is "incomplete vs blocked" —
   particularly for budget-exhausted cases that don't fit
   DEPTH_EXCEEDED.
5. **Cross-rule audit detail.** The audit reports a single matched
   rule. With v1.7's universal-conclusion branch sharing the
   `SYLLOGISM` label, downstream consumers cannot distinguish
   particular-Barbara from universal-Barbara. v1.8 could expose
   a sub-form tag on the `InferenceMatch`.
6. **Stem-aware predicate matching.** v1.6's narrow inflection
   table (5 regular plurals + 6 irregular) handles the Cat-B
   noun-morphology cases. v1.8 may need verb-form matching for
   negations to enable contradiction across "is not happy" and
   "happy" — currently those don't collide.
7. **Audit-error ↔ blocking-reason coupling.** The v1.7
   classifier reads `audit.gap.kind` directly. A future change
   to `GapKind` would silently change classification. v1.8 should
   make the mapping a dispatch table that fails-loud on an
   unknown kind.
8. **Per-source-document blocking-reason aggregation.** Multi-claim
   documents will eventually produce many resolution results.
   v1.8 may want a per-document aggregator that surfaces the
   *dominant* blocking_reason (and named outliers) so reviewers
   can spot parser regressions quickly.
9. **The Cat-D and Cat-E parser blocks.** Most Cat-D / Cat-E
   blocks now correctly carry COUNTEREXAMPLE_FOUND or
   PARSER_UNSUPPORTED_FORM. v1.8 should think about whether a
   genuine philosophical leap (E10 cosmological argument) should
   produce a *third* category like "structurally well-formed but
   unsupportable" — beyond what the v1.7 enum captures.
