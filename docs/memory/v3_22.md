### Key Observations

1. **ATTACK_SUCCESS_RATE is identical across all 8 orderings.**
   Each ordering produces the same 0.195 — the set of
   v3.16-surviving cases is determined by the SuspensionGate's
   per-chain predicate, not by the gate sequence. Order does
   not change which chains slip through; it only changes
   *which gate* fires first for each chain.

2. **`maximal_safe_stack` has the lowest latency_cost (3.70)
   and lowest false_route_count (85).** Putting
   SuspensionGate early (right after parsing) makes most
   adversarial chains terminate sooner, reducing the average
   gate-fire index. This is the only ordering that
   demonstrably improves on the current order in *latency* —
   but not in attack_success_rate.

3. **`minimal_without_causal_chain` has the highest
   heldout_recall (0.893).** When G6 is removed, valid chains
   that previously got blocked by CAUSAL_CHAIN's mismatch
   guards now pass through. But this same removal lets
   adversarials reach the SuspensionGate at the same rate.
   It's a recall win without a precision cost — but valid
   chains are no longer *supported via CAUSAL_CHAIN*, only
   *allowed to pass*, which is a downgrade of the verdict's
   meaning.

4. **All eight orderings fail at least one hard gate.** The
   v3.16 surviving subset (85 cases) is by construction the
   adversarial residual that no ordering of the current gate
   set can eliminate. Recall is dominated by the
   FrameTension's UNDECIDABLE verdicts on context-less
   valid chains (a corpus-construction artefact: the test
   chains lack inherited_context, so FrameTension cannot
   produce CONFIRMED for them).

## Reflexion

> *"A gate is only a defence if it fires before the damage
> propagates."*

**Empirisch: Die Reihenfolge der Gates ändert *wann* sie
feuern, aber nicht *welche* Chains durchkommen. Das v3.16-
surviving Set (85 Cases) ist ein per-chain Phänomen —
SuspensionGate erkennt es nicht, also schlüpft es durch,
ganz egal in welcher Position SuspensionGate im Stack steht.
Die einzige meßbare Order-Wirkung ist auf die *Latency*
(wann der Block fällt) und den *false_route_count* (wie
viele Adversarials durchs Routing rutschen, bevor sie
geblockt werden).**

### Was die Daten zeigen

| Beobachtung | Quelle |
| --- | --- |
| 8 Orderings, alle mit ASR = 0.195 | per-ordering metrics |
| Best latency_cost = 3.70 (maximal_safe_stack) | metrics |
| Best heldout_recall = 0.893 (minimal_without_causal_chain) | metrics |
| Best false_route_count = 85 (maximal_safe + causal_before_frame_tension) | metrics |
| Kein Ordering schlägt baseline auf ASR | report |
| Empfehlung: KEEP_CURRENT_ORDER | report |

### Drei Lesarten der Schwelle

1. **Order moves latency, not effectiveness.** Die Gates
   wirken per-chain unabhängig: SuspensionGate blockt
   bestimmte Marker-Chains, FrameTension blockt bestimmte
   Mismatch-Chains, CAUSAL_CHAIN blockt non-Chain-Shapes.
   Welcher zuerst feuert, ändert die *Reihenfolge der
   Verdikte*, nicht das *Ergebnis*. Das ist eine direkte
   empirische Bestätigung der Suspension-Architektur: jeder
   Gate ist ein unabhängiger Verweigerer.

2. **`maximal_safe_stack` ist die latency-optimal Ordering
   ohne ASR-Verbesserung.** Putting SuspensionGate als
   Gate 3 (statt Gate 7) macht 388 → 85 false_routes (die
   meisten Adversarials bekommen ein Frühblock). Das spart
   Pipeline-Arbeit, aber rettet keinen Adversarial der vorher
   nicht gerettet wurde. Eine reine Latency-Optimierung.

3. **`minimal_without_causal_chain` ist ein interessantes
   Datum.** Recall steigt 0.649 → 0.893 wenn G6 entfernt wird.
   Das heißt: CAUSAL_CHAIN blockt ~25% der validen Chains
   im aktuellen Setup. Ohne G6 würden sie alle als
   PASSED durchlaufen — aber ohne Inference-Verdikt
   (kein "LOGICALLY_SUPPORTED"). Das ist eine Verlagerung
   der Last vom Gate auf den Downstream-Reader.

### Die operative Antwort

> **Ein Gate ist nur dann eine Defense wenn es vor der
> Schaden-Propagation feuert. Aber die v3.22-Daten zeigen:
> in der aktuellen DESi-Konfiguration sind alle sechs
> Cliffs *gleich effektiv* unabhängig von ihrer Position.
> Das verschieben der Reihenfolge ändert den Latency-
> Profil, nicht die Schutzwirkung. Der Schutz ist eine
> per-chain Eigenschaft der einzelnen Gates, nicht eine
> emergente Eigenschaft der Sequence.**

### Drei Konsequenzen

1. **Reorder lohnt sich nicht für Sicherheit, aber für
   Latency.** Die `maximal_safe_stack`-Ordering hat die
   gleiche Sicherheit wie das current_order, aber spart 30 %
   Latency (3.70 vs 5.21 average fire-index). Wenn
   Performance ein Concern wäre, wäre Reorder eine
   sinnvolle Option. Sicherheit ist es nicht.

2. **`minimal_without_causal_chain` ist gefährlich
   missverständlich.** Höhere Recall = "mehr Chains kommen
   durch". Das klingt wie "DESi wird durchlässiger". In
   Wirklichkeit: sie kommen durch *ohne Verdict*. Der
   Recall-Gewinn ist nicht Inferenz-Gewinn, sondern
   Inferenz-Vermeidung.

3. **`minimal_without_frame_tension` zeigt G4 ist
   kritisch für Routing.** Ohne FrameTension fallen Recall
   und ASR auf gleiches Niveau wie current_order — aber
   contamination steigt nicht. Das ist kompatibel mit v3.21's
   Befund (G4 ist PRIMARY_CLIFF auf contamination_count),
   denn ohne G4 routet der Router weniger restriktiv. v3.22
   misst diese Wirkung in der Order-Dimension.

### Was nicht behauptet wird

* **Nicht:** Die aktuelle Reihenfolge ist beweisbar optimal.
  Sie ist *nicht schlechter* als die getesteten Alternativen.
  Es könnte Reihenfolgen geben, die wir nicht getestet
  haben, die besser sind.
* **Nicht:** Order ist semantisch egal. Die Latency-
  Unterschiede (5.31 vs 3.70) sind real und in
  performance-kritischen Pfaden bedeutsam.
* **Nicht:** Die hard-gate failures sind ein Pipeline-Versagen.
  Sie sind eine Korpus-Konstruktion: v3.16-surviving Cases
  sind per Definition Adversarials die kein bekannter Gate
  fängt, und die Heldout-Recall-Probleme stammen aus
  context-leeren Test-Chains, nicht aus echten
  Production-Routings.

> **Die Reihenfolge der Gates verschiebt das Timing der
> Verteidigung, nicht ihre Effektivität. Die aktuelle
> Reihenfolge ist nicht beweisbar optimal — aber kein
> getestetes Alternativ schlägt sie auf
> attack_success_rate. Damit ist KEEP_CURRENT_ORDER die
> ehrliche datenbasierte Empfehlung. Wer Latency
> optimieren will, hat einen messbaren Kandidaten
> (maximal_safe_stack). Wer Sicherheit optimieren will,
> findet keinen Reorder-Hebel.**
