# DESi v0.8 — Multi-Seed Significance

v0.8 is **the reproducibility release**. Up to v0.7, a single
deterministic seed was enough to promote a mutation. From v0.8 on,
that is no longer sufficient: promotion requires that the
improvement signal holds across a closed list of seeds.

Design principle:

> Ein einzelner Erfolg ist eine Beobachtung.
> Mehrere reproduzierbare Erfolge sind Evidenz.

## The five mandatory seeds

```
seeds = (42, 43, 44, 45, 46)
```

The seed list is hard-coded, fixed, deterministic. v0.8 explicitly
forbids random seeds. A multi-seed report whose seed list does not
match this tuple is rejected by the gate as ``inconclusive``.

The optional stress seed `999` is run **only** for
`ADV_BRANCH_EXPLOSION` and is **not** promotion-relevant. It is
logged on the report for forensics.

## MultiSeedEvaluationReport

```python
@dataclass
class MultiSeedEvaluationReport:
    mutation_id: str
    clone_id: str
    parent_version: str
    timestamp: datetime
    scenario_ids: tuple[str, ...]
    seeds: tuple[int, ...]
    per_seed_reports: tuple[PairedEvaluationReport, ...]
    aggregates: dict[str, ScenarioAggregate]
    stress_outcome: PairedScenarioOutcome | None
```

`aggregates[scenario_id]` exposes:

| Field                  | Type   | Notes                            |
|---                     |---     |---                                |
| `n_seeds`              | int    | always 5 for the default runner   |
| `mean_branch_delta`    | float  | `statistics.fmean` of per-seed Δ  |
| `median_branch_delta`  | float  | `statistics.median`               |
| `std_branch_delta`     | float  | population std (`pstdev`)         |
| `mean_timeline_delta`  | float  | secondary signal                  |
| `mean_guard_delta`     | float  | mechanism-of-action signal        |
| `per_seed_verdicts`    | tuple  | one verdict string per seed       |
| `improved_seed_count`  | int    | seeds verdict==improved           |
| `neutral_seed_count`   | int    | seeds verdict==neutral            |
| `regressed_seed_count` | int    | seeds verdict==regressed          |

Aggregate verdict on the full report:

| Verdict         | Rule                                              |
|---              |---                                                |
| `regressed`     | any scenario has any seed verdict==regressed      |
| `neutral`       | every per-seed verdict is neutral                 |
| `improved`      | strongest candidate has ≥4/5 improved seeds       |
| `inconclusive`  | some improvement but not robust                   |

## SignificanceGate

```python
SignificanceGate(
    required_seed_count=5,
    improved_threshold=4,
).decide(report) -> SignificanceDecision
```

Rules:

1. **Any seed regressed on any guard (S2 / S6)** → `regressed`.
2. **Any seed regressed on the candidate scenario** → `regressed`.
3. **Strongest candidate ≥4/5 improved AND zero regressions** →
   `improved`.
4. **No candidate improved on any seed** → `neutral`.
5. **Some improvement on a candidate, but <4/5** → `inconclusive`.

`SignificanceDecision` is frozen:

```python
@dataclass(frozen=True)
class SignificanceDecision:
    mutation_id: str
    verdict: str          # improved | neutral | regressed | inconclusive
    supporting_seeds: tuple[int, ...]
    failing_seeds: tuple[int, ...]
    rationale: str
```

## Jury extension

v0.7 added `paired_report` kwarg. v0.8 adds `multi_seed_report`:

* `multi_seed_report` **supersedes** `paired_report` for the
  Integrator role. When supplied, the Integrator runs the
  SignificanceGate and votes APPROVE / REVISE / VETO accordingly.
* The Skeptiker probes robustness: <4/5 improved on the candidate
  → REVISE, regardless of regression status.
* The Replikator (the variance role, "Empiriker" in the directive)
  probes variance: `std_branch_delta > 1.0` on any aggregate →
  REVISE.

v0.7 and v0.6 callers that omit `multi_seed_report` see the same
decision as before. Verified by
`test_jury_without_multi_seed_report_matches_v07_signature`.

## Ledger extension

Three new append-only event types:

```python
LedgerEventType.MULTI_SEED_STARTED     # mutation_id, scenario_ids, seeds
LedgerEventType.MULTI_SEED_RESULT      # scenario_id, aggregated_metrics,
                                       # verdict
LedgerEventType.SIGNIFICANCE_DECISION  # mutation_id, verdict,
                                       # supporting_seeds, failing_seeds
```

A v0.8 promotion cycle writes one `MULTI_SEED_STARTED`, one
`MULTI_SEED_RESULT` per scenario, and exactly one
`SIGNIFICANCE_DECISION`. All three round-trip through the JSONL
ledger with deterministic `payload_hash`.

## M-001 robustness — was the v0.7 promotion seed-luck?

```
M-001 over seeds (42, 43, 44, 45, 46):

  S5                       improved=0/5  std=0.0   neutral
  ADV_BRANCH_EXPLOSION     improved=5/5  std=0.0   robust
  S2                       improved=0/5  std=0.0   neutral
  S6                       improved=0/5  std=0.0   neutral

aggregate_verdict        improved
SignificanceGate         verdict='improved'
  supporting_seeds       (42, 43, 44, 45, 46)
  failing_seeds          ()
  rationale              candidate scenario ADV_BRANCH_EXPLOSION
                         cleared 5/5 improved seeds (threshold 4/5);
                         every regression guard 5/5 neutral-or-improved.

Stress seed 999          ADV_BRANCH_EXPLOSION verdict='improved'
```

The result is **robust by the v0.8 gate**: every seed lands on the
same per-scenario verdict, branch reduction holds at exactly −1 on
all five seeds, and no regression guard ever fires. The v0.7
promotion was not seed-luck — but the deeper reason is informative:
the v0.7 branch-evidence function is fully deterministic on the
trajectory structure and does not depend on the seed value. v0.8's
infrastructure is in place; v0.9 will need scenarios with
seed-driven variance to make the multi-seed gate consequential
(see open question 1 below).

## What v0.8 deliberately does NOT do

* no real LLM API calls
* no internet access
* no Memory reads inside a run
* no second activated config knob
* no parallel mutation scheduling
* no automatic Proposal generation
* no Neo4j ledger backend (JSONL only)

The directive: **same mutation. more evidence. less self-deception.**

## Open design questions for v0.9

1. **Scenarios with real seed-driven variance.** v0.7's mechanism is
   deterministic on the trajectory structure, so the v0.8 multi-seed
   gate currently observes `std=0.0` on every aggregate. v0.9 should
   introduce scenarios whose claim ordering, focus selection, or
   merge eligibility depends on the seed — otherwise the gate's
   variance probe is decorative.
2. **Bootstrap confidence intervals.** v0.8 uses a simple "≥4/5
   improved" threshold. v0.9 could add bootstrap resampling over the
   seed set and require that the 95% CI of the mean branch delta lies
   strictly below zero.
3. **Per-mutation seed budget.** Right now every mutation pays for 5
   full paired evaluations (10 runs × 4 scenarios). Larger candidate
   sets will need a budget mechanism — and a rationale for the budget
   that does not silently weaken the gate.
4. **Variance threshold as a function of N.** The Empiriker's
   `std > 1.0` REVISE threshold is hand-set. Once real variance
   exists, the threshold should be derived from the candidate's
   baseline noise floor (e.g. via S5 std on stable runs).
5. **Stress-seed escalation.** Seed 999 currently logs without
   feedback. v0.9 could promote it to a soft veto if its outcome
   differs from the default seed set by more than a configured
   margin.
6. **Per-seed JSONL detail records.** Today the ledger logs
   per-scenario aggregates; v0.9 should consider one
   `MULTI_SEED_RESULT` per (scenario, seed) pair so the audit log
   can reconstruct individual runs without re-running.
7. **Inconclusive feedback loop.** REVISE without a structured next
   action risks an endless loop. v0.9 needs an explicit
   `RevisionRequest` artifact: "what specifically must change before
   re-evaluating?"
8. **Real-LLM jury rehearsal under multi-seed.** Once a real-LLM
   persona is introduced, "seed" gains a second meaning (sampling
   temperature seed). The Empiriker variance check must distinguish
   model-seed from scenario-seed.
