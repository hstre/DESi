# DESi v0.7 — Config Activation + Metrics Delta

v0.7 is **the first behaviour-effective evolution release**. For the
first time, a configured mutation actually changes how DESi runs.
The change is deliberate, narrow, fully auditable, and gated by a
measurable verdict.

Design principle:

> Eine Mutation ist erst real, wenn sie das Verhalten verändert.
> Eine Verhaltensänderung ist erst gültig, wenn sie messbar besser
> ist.

## The one activated knob

`guard_thresholds.branch_open_evidence_min` — a real-valued threshold
that gates branch opening. The branch-evidence function is
deterministic and depends only on how many distinct foci have already
opened branches in the current session:

```
evidence(i) = max(0.0, 0.7 - 0.10 * opens_so_far(i))
```

When `config=None` (every v0.6 call), the threshold defaults to 0.0
and every focus opens a branch — bit-identical to v0.6 (tested).

When `config={"guard_thresholds.branch_open_evidence_min": 0.30}`
(stable default), the threshold is 0.30. On ADV_BRANCH_EXPLOSION
this suppresses the last two branches (4 opens + 2 guard_blocked).

When `config={"guard_thresholds.branch_open_evidence_min": 0.45}`
(M-001 mutation), three opens + three guard_blocked.

Every branch_opened / guard_blocked event carries `evidence` and
`threshold` in its payload — no hidden parameters.

## MetricsDelta

```python
@dataclass(frozen=True)
class MetricsDelta:
    stable: PathQualityMetrics
    clone: PathQualityMetrics
    # absolute
    @property
    def timeline_length_delta(self) -> int
    def branch_opened_delta(self) -> int
    def guard_blocked_delta(self) -> int
    def contradicts_delta(self) -> int
    def merged_into_delta(self) -> int
    def hook_error_delta(self) -> int
    # relative
    def branch_reduction_pct(self) -> float
    def timeline_reduction_pct(self) -> float
    # verdict
    @property
    def verdict(self) -> str  # "improved" | "neutral" | "regressed"
```

Verdict rules (v0.7):

* `improved`  — branches reduced AND no other regression
* `regressed` — branches increased OR contradicts drifted OR
                merged_into drifted OR hook errors increased OR
                timeline lengthened beyond branch/guard accounting
* `neutral`   — every delta is zero

`guard_blocked_delta` is treated as **neutral** in v0.7 because the
only behaviour-effective knob generates these events as its mechanism
of action. Counting them as regressions would mark every working
mutation as a regression.

## Paired evaluation pipeline

```
stable_config   (clone.stable.as_dict)
       │
       ▼
[stable run on scenario]   →   PathQualityMetrics(stable)
       │
       ▼                       (deterministic raw counts)
clone_config (clone.config)
       │
       ▼
[clone run on scenario]    →   PathQualityMetrics(clone)
       │
       ▼
MetricsDelta(stable, clone)
       │
       ▼
PairedScenarioOutcome
       │
       ▼                       (one outcome per scenario)
PairedEvaluationReport
       │
       ▼
DelphiJury.deliberate(..., paired_report=...)
       │
       ▼
PromotionGate → promote / VETO / REVISE
```

v0.7 scenario set (closed, hard-coded):

| Role                     | Scenario(s)                  |
|---                       |---                            |
| Evolution candidate      | `S5`                         |
| Pflicht adversarial      | `ADV_BRANCH_EXPLOSION`       |
| Regression guards        | `S2`, `S6`                   |

A clone NEVER promotes on a single scenario; the paired evaluation
always covers all four.

## JSONL ledger

```python
from desi.evolution import EvolutionLedgerJSONL, LedgerEventType

ledger = EvolutionLedgerJSONL("ledger.jsonl")
ledger.append(LedgerEventType.PROPOSAL, {"mutation_id": "M-001"})
ledger.append(LedgerEventType.CONFIG_ACTIVATED, {
    "mutation_id": "M-001",
    "clone_id": "clone_a3c8…",
    "config_hash": "deadbeef",
    "active_knobs": ["guard_thresholds.branch_open_evidence_min"],
})
ledger.append(LedgerEventType.METRICS_DELTA, {
    "scenario_id": "ADV_BRANCH_EXPLOSION",
    "verdict": "improved",
    "branch_opened_delta": -1,
    "guard_blocked_delta": +1,
})
ledger.append(LedgerEventType.EVOLUTION_CYCLE, {
    "mutation_id": "M-001",
    "aggregate_verdict": "improved",
})
```

Properties:

* One JSON object per line. Each line is canonical-JSON (sort_keys,
  no whitespace) so two ledgers given the same payload produce
  byte-identical lines.
* `payload_hash` is deterministic — two processes appending the same
  payload get the same 16-char hash.
* Re-opening an existing JSONL replays all prior entries into the
  in-memory ledger without touching the file.
* The file is append-only at the disk level: open mode is always
  `"a"`, never `"w"`. Tests verify the file size only grows.

## Jury extension

`DelphiJury.deliberate` accepts an optional `paired_report` kwarg. The
Integrator role consults it in round 2:

* `aggregate_verdict == "improved"` → APPROVE with rationale citing
  the measurable improvement
* `aggregate_verdict == "neutral"` → REVISE ("no measurable
  improvement yet")
* `aggregate_verdict == "regressed"` → VETO with a structured
  `Veto(affected_claim, suspected_risk, failure_case, proposed_test)`

v0.6 callers that omit `paired_report` see the same decision they
would have seen before (the Integrator only invokes the new logic
when a report is attached).

## What v0.7 deliberately does NOT do

* no real LLM API calls
* no internet access
* no Memory reads inside a run
* no additional config knobs beyond `branch_open_evidence_min`
* no automatic Proposal generation
* no parallel-clone scheduling
* no Neo4j ledger backend (JSONL only)
* no statistical significance tests
