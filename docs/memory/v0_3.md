# DESi memory layer — v0.3

v0.3 introduces the **MemoryHook**: a write-only observer that sits
outside the DESi run loop and writes everything DESi produces to a
memory store. The hook does not read from memory during the run, does
not influence the run's verdicts, and is optional in the strict sense
of the word — DESi continues to work without it on a host without
Neo4j, without the `neo4j` driver, and without any database at all.

Design principle: *Der Hook beobachtet den Run. Die Operatoren bleiben
rein.*

## New surface

```
src/desi/runner.py       run_desi(trajectory, *, memory_hook=None)
src/desi/memory/hook.py  MemoryHook + HookError
```

The entrypoint `run_desi` is a thin wrapper around `compute_all`:

```python
from desi.runner import run_desi

# 1. The unchanged path. Same behaviour as pre-v0.3.
report = run_desi(trajectory)

# 2. Opt-in observation. The hook writes to its recorder; the
#    returned report is identical to the no-hook case.
from desi.memory import InMemoryStore, MemoryRecorder, MemoryHook
hook = MemoryHook(MemoryRecorder(InMemoryStore()),
                  model="claude-opus-4-7",
                  config={"seed": 7})
report = run_desi(trajectory, memory_hook=hook)
```

`run_desi(trajectory) == run_desi(trajectory, memory_hook=hook)` for
every trajectory — this is asserted by `tests/test_runner.py`.

## What the hook records

For every run handed to the runner the hook writes:

| Memory record   | When                                    | Source field                           |
|---              |---                                      |---                                     |
| `Run`           | `on_run_start` (once)                    | `trajectory.trajectory_id`             |
| `OperatorEvent` | `on_step` (once per `TrajectoryStep`)   | `step.operator`, `step.loop_index`     |
| `Claim`         | `on_step` (first time a focus is seen)  | `step.focus_claim_id`                  |
| `DERIVES_FROM`  | `on_step` (when focus changes)          | `(prev_focus, new_focus)`              |
| `Run.finished_at` | `on_run_end`                            | wall clock                             |

There are deliberately no diagnostic verdicts written by v0.3 — the
hook records the trajectory, not the report. This stays inside the
"observe, don't interpret" v0.3 directive.

## Error isolation

The hook wraps every recorder call in a `_safe` shim. Two modes:

```python
# Default. A failing recorder is captured on hook.errors;
# the run continues and produces an identical report.
hook = MemoryHook(rec, model="m")
report = run_desi(trajectory, memory_hook=hook)
assert hook.errors  # populated if writes failed
assert report       # produced regardless

# Strict. A failing recorder re-raises;
# the run aborts.
strict = MemoryHook(rec, model="m", strict=True)
run_desi(trajectory, memory_hook=strict)  # may raise
```

Every captured failure is a `HookError(stage, exception, payload)` so
that downstream callers can tell **which** lifecycle stage broke and
**which** step or claim was involved.

## What v0.3 does NOT change

Enumerated from the v0.3 directive:

- Operator selection — untouched.
- Guard results — untouched.
- Branch logic — untouched.
- Merge logic — untouched (still v0.2's atomic helper).
- Confidence values — untouched.
- Report generation — untouched.
- Diagnostic verdict — untouched.

The runner does not read from `ReadOnlyMemoryView` during the run.
`ReadOnlyMemoryView` exists and is tested, but it is reserved for
later, deliberately-staged experiments. Bringing it into the run is a
v0.4 question, not v0.3 plumbing.
