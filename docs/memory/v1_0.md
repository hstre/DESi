# DESi v1.0 — SPL DESi Adapter

v1.0 is **the first sensory release**. Until v0.9, every DESi claim
was authored by a controlled trajectory or a hand-crafted scenario.
v1.0 introduces the first path from **free-form natural-language
text** to a DESi Claim — through a strict pipeline of backend,
gateway, and audited mapping.

Design principle:

> The NLP system is a sensor.
> Not a judge.
> Not a memory.
> Not a merge instance.
> Not a promotion instance.
> Not a mutation instance.

## The pipeline

```
Text
  ↓
SemanticBackend.extract_units(text)            # deterministic OR LLM
  ↓
SemanticBackend.project_units(units, …)        # → tuple[ClaimCandidate, …]
  ↓
SPLGateway.admit(candidates, run_id=…)         # closed rejection taxonomy
  ↓
candidate_to_claim(c, run_id=…)                # → desi.memory.claim.Claim
  ↓
SPLProjectionResult.claims  (caller decides whether to persist)
```

The adapter never writes memory, never decides merges, never decides
promotion. The return value is just a tuple of Claim objects.

## The two backends

### `DeterministicSemanticBackend`

Mandatory, no APIs, fully reproducible. Ships with a deliberately
narrow phrase library (the point is a deterministic baseline for the
test suite, not real-world NLP). Recognises the phrase family used by
the P1 paraphrase test:

```
"water boils at <N>"
"water reaches boiling point at <N>"
"boiling point at <N> degrees"
```

…and canonicalises every match to ``"water boils at <N>°C"``. Hedge
words (``maybe``, ``perhaps``, ``might``, ``possibly``, ``uncertain``,
``unclear``) flag the unit as ``ambiguous=True`` so the gateway can
reject it.

`method_label = "deterministic_semantic_projection"`.

### `LLMSemanticBackend`

Optional. **Disabled by default.** v1.0 hard-codes the model to
`deepseek-4-pro` — switching to another model requires a code edit so
that the audit trail cannot silently drift.

The backend never calls the network on its own. It calls the injected
``llm_call: Callable[[str], str]`` callable. Real network code is the
caller's responsibility; the test suite drives every fail-closed path
through callables that return malformed JSON, raise `TimeoutError`,
return empty strings, etc.

Response schema (strictly required):

```json
{"units": [
  {"canonical_content": "...",
   "raw_span": "...",
   "confidence": 0.0..1.0,
   "ambiguous": true|false,
   "proposed_relations": []}
]}
```

Any deviation → `BackendError` → adapter fail-closes (zero claims,
ledger entry).

`method_label = "llm_semantic_projection"`.

## SPLProvenance — the rich audit record

The four-field DESi `Provenance` stays as it is. SPL carries a richer
`SPLProvenance` alongside each `ClaimCandidate`:

| Field           | Mandatory for `spl_llm` | Notes                          |
|---              |---                      |---                              |
| `source`        | always                  | `spl_deterministic` / `spl_llm` |
| `model_name`    | yes                     | hard-coded `deepseek-4-pro`     |
| `model_version` | yes                     | passed at construction          |
| `prompt_hash`   | yes                     | `ph_` + sha256-16 of prompt     |
| `temperature`   | yes                     | float                           |
| `max_tokens`    | yes                     | int                             |
| `timestamp`     | yes                     | UTC datetime                    |
| `output_hash`   | yes                     | `oh_` + sha256-16 of output     |
| `document_id`   | optional                | caller-supplied                 |
| `author`        | optional                | caller-supplied                 |
| `language`      | optional                | caller-supplied                 |

`SPLProvenance.is_complete_llm_record` returns True iff every
mandatory LLM field is populated.

When `candidate_to_claim()` maps a candidate to a `Claim`, it copies
the four DESi-required fields from `SPLProvenance` into the Claim's
`Provenance`. The LLM-specific fields are recorded in the
`SEMANTIC_CANDIDATE_EMITTED` ledger event so the audit trail keeps
everything the SPL saw.

## SPLGateway — the closed rejection taxonomy

The gateway is the only place where a `ClaimCandidate` becomes a
`Claim`. Rejection reasons are a closed set:

| Reason                    | Trigger                                      |
|---                        |---                                            |
| `empty_content`           | `candidate.content == ""`                     |
| `invalid_method`          | `method` not in `LEGAL_METHODS`               |
| `ambiguous_claim`         | `ambiguous=True` OR `confidence < 0.5`         |
| `hallucinated_relation`   | `proposed_relations` is non-empty             |
| `no_backend_output`       | backend returned zero candidates              |
| `backend_error:<kind>`    | any `BackendError`                            |
| `cost_guard_exhausted`    | LLM budget hit                                |

The gateway is observable: rejected candidates are returned alongside
admitted claims as `(candidate, reason)` pairs, and each rejection
emits a `SEMANTIC_PROJECTION_REJECTED` ledger event when a ledger is
wired in.

## Mapping invariants

`candidate_to_claim` enforces three v1.0 contracts:

1. **State always `PROPOSED`.** SPL never marks a Claim CONFIRMED,
   REVISED, MERGED, REJECTED, or SPLIT.
2. **Method always one of two strings.** Anything outside
   `LEGAL_METHODS = {"deterministic_semantic_projection",
   "llm_semantic_projection"}` is a gateway error.
3. **Content ≠ method.** Two claims with the same canonical content
   but different methods produce distinct `claim_id`s because
   `Claim._derive_id` hashes `content + method + run_id`. The S7
   memory-trap invariant is preserved end-to-end.

## Cost guard

```python
LLM_PROJECTION_BUDGET = 20  # per SPLAdapter instance
```

The 21st `project_text(…)` call on an `LLMSemanticBackend`-backed
adapter is **fail-closed**: it returns zero claims with
`result.cost_guarded == True` and emits a
`SEMANTIC_PROJECTION_REJECTED { reason: "cost_guard_exhausted" }`
ledger event. `llm_calls_made` is not incremented when the guard
fires (the budget is not over-spent in audit terms). The
deterministic backend is never cost-guarded.

The budget is per-adapter, so two adapters do not share state.

## Fail-closed semantics

Every failure path produces **zero claims** plus a ledger event:

| Failure                       | Result                                            |
|---                            |---                                                 |
| `TimeoutError` in `llm_call`  | `backend_error="llm_call_failed"`, `claims=()`     |
| `ConnectionError`             | `backend_error="llm_call_failed"`, `claims=()`     |
| Non-JSON output               | `backend_error="invalid_json"`, `claims=()`        |
| Missing `units` key           | `backend_error="invalid_schema"`, `claims=()`      |
| `units` not a list            | `backend_error="invalid_schema"`, `claims=()`      |
| Empty output                  | `backend_error="empty_output"`, `claims=()`        |
| Empty unit list               | `claims=()` (not an error, just no signal)         |
| Cost guard hit                | `cost_guarded=True`, `claims=()`                   |

No backend exception ever crosses the adapter boundary. The adapter
is the firewall between fragile NLP and the rest of DESi.

## Example: P1 (paraphrase)

```python
adapter = SPLAdapter()
a = adapter.project_text("Water boils at 100C.",
                          run_id="run_p1")
b = adapter.project_text(
    "At one atmosphere water reaches boiling point at 100 degrees.",
    run_id="run_p1",
)

a.claims[0].content      == "water boils at 100°C"
b.claims[0].content      == "water boils at 100°C"
a.claims[0].claim_id     == b.claims[0].claim_id   # same content +
                                                    # same method +
                                                    # same run_id
```

## Example: P2 (same content, different method)

```python
det_adapter = SPLAdapter()
llm_adapter = SPLAdapter(backend=LLMSemanticBackend(llm_call=fn))

det = det_adapter.project_text("Water boils at 100C.", run_id="run_p2")
llm = llm_adapter.project_text("Water boils at 100C.", run_id="run_p2")

det.claims[0].content  == llm.claims[0].content
det.claims[0].method   == "deterministic_semantic_projection"
llm.claims[0].method   == "llm_semantic_projection"
det.claims[0].claim_id != llm.claims[0].claim_id   # S7 invariant preserved
```

## Example: gateway reject

```python
adapter = SPLAdapter()
result = adapter.project_text(
    "Water might possibly boil somewhere around 100 degrees."
)
result.claims                       == ()
result.rejected[0][1]               == "ambiguous_claim"

# A LLM that proposed a SUPPORTS relation:
result.rejected[0][1]               == "hallucinated_relation"
```

## Example: cost guard

```python
adapter = SPLAdapter(backend=LLMSemanticBackend(llm_call=fn))
for _ in range(20):
    adapter.project_text("Water boils at 100C.")    # all admitted
result = adapter.project_text("Water boils at 100C.")
result.claims        == ()
result.cost_guarded  == True
adapter.llm_calls_made           == 20   # budget not over-spent
adapter.llm_calls_remaining      == 0
```

disk. Re-opening replays all entries (v0.7 JSONL contract).

## Did SPL integrate cleanly into DESi?

**Yes.** The full v0.9 baseline (434 tests) is unchanged. The SPL
package lives in its own `src/desi/spl_adapter/` directory, exports
only its public surface, and never touches memory / observe /
evolution state. The only cross-cutting change is the 3-event
extension to `LedgerEventType` (`SEMANTIC_PROJECTION_*`).

A v0.9 caller that never imports `desi.spl_adapter` runs exactly as
it did in v0.9. A v1.0 caller that uses `SPLAdapter()` gets a tuple
of `Claim` objects whose `method` is one of two legal values and
whose `state` is always `PROPOSED` — feeding them into the
`MemoryRecorder` follows the same path as any other claim source.

## What v1.0 deliberately does NOT do

* no real promotion through an LLM
* no Memory reads during a run
* no LLM merge decisions
* no LLM jury
* no automatic mutation
* no second mutation knob
* no internet calls outside the explicit DeepSeek path
* no fallback to other models

The directive: **semantics as sensor, not as authority.**

## Open design questions for v1.1

1. **Real DeepSeek call site.** v1.0 ships the `LLMSemanticBackend`
   with a default `llm_call` that fail-closes. v1.1 needs a real
   client adapter (with timeout, retry caps, response-size limits,
   Vault-only key handling, and rate limiting). The client must be a
   thin shim so that *all* the validation, cost-guarding, and
   fail-closing stays inside `SPLAdapter`.
2. **Multi-modal input.** v1.0 takes plain `str`. v1.1 should
   formalise a `SourceDocument` type with content + content-type +
   author + document_id, so that PDF, HTML, and email sources don't
   silently flatten into `str`.
3. **Cross-source merge guard.** Two SPL claims with the same content
   from two different documents are currently distinct (different
   `run_id`s). v1.1 needs a policy for whether DESi's merge operator
   should propose a merge across documents — and if so, with what
   evidence.
4. **Phrase library scope.** The v1.0 deterministic backend
   recognises one phrase family (`water boils at X`). Production use
   requires a controlled vocabulary. The question for v1.1: does the
   deterministic backend grow, or does it stay as the test baseline
   while a richer rule engine (built independently) ships under its
   own method label?
5. **Confidence floor as a config knob.** `AMBIGUITY_FLOOR = 0.5` is
   currently hard-coded. Once a second behaviour-effective config
   knob is allowed (v1.1 design constraint), this is a natural
   candidate — but the directive's "second knob" reservation may push
   it further.
6. **Per-document cost budget.** Today the budget is per-adapter.
   v1.1 may need a per-`document_id` budget so a single large
   document cannot exhaust a session's projection allowance.
7. **Backend-version pinning across ledger replays.** Replaying a
   v1.0 ledger that contains `model_name=deepseek-4-pro` on a future
   build with a different default model must either reject the
   replay or surface a warning. v1.1 should formalise the
   compatibility contract.
8. **LLM hallucination beyond relations.** v1.0 blocks only
   `proposed_relations`. v1.1 should consider other hallucination
   signatures: unit-of-measurement drift, content that contradicts
   the raw_span, content with placeholder tokens, etc.
9. **Jury awareness of SPL provenance.** The v0.6+ jury does not
   know that a Claim came through SPL. v1.1 could let the Skeptiker
   downweight evidence drawn from `spl_llm` until cross-validation
   from `spl_deterministic` or a primary source is recorded.
