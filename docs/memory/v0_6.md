# DESi v0.6 — Evolution Ledger + Veto-to-Test Synthesiser

v0.6 hardens the v0.5 constitutional process with two load-bearing
additions:

* **EvolutionLedger** — append-only audit trail covering every step
  of Reflection → Proposal → Clone → Evaluation → Jury → Promotion.
* **VetoToTestSynthesiser** — every valid veto turns into a structured
  `VetoTestObligation` plus a runnable pytest skeleton. Promotion is
  blocked until each obligation reaches a terminal state (PASSED or
  WAIVED with rationale).

Design principle:

> Keine Evolution ohne forensischen Audit-Trail.
> Kein Veto ohne daraus folgende Prüfung.

v0.6 does not change DESi behaviour. No real LLM calls, no internet
access, no behaviour-effective config_delta. The work in this release
is auditability and operationalisation of vetos.

## New surface

```
src/desi/evolution/

  veto_followup.py   VetoToTestSynthesiser + VetoTestObligation
                     + ObligationStatus
  metrics.py         PathQualityMetrics + compute_path_quality
  draft.py           ProposalDraftBuilder (requires_ratification=True)
```

Existing modules updated (additive only):

```
proposal.py     + requires_ratification: bool = False
jury.py         + JuryMember.model_label
reflection.py   + ReflectionEngine.reflect(..., memory_view=None)
                + cross-run recurrence detector (memory-aware,
                  post-run only)
promotion.py    + promote_with_ledger(...)
                + ledger completeness checks
                + open-obligation gating
```

## EvolutionLedger

Closed event-type enum (13 kinds):

```
REFLECTION, PROPOSAL, CLONE_CREATED, EVALUATION,
JURY_ROUND1, JURY_ROUND2,
VETO_VALID, VETO_INVALID,
PROMOTION_DECISION, SNAPSHOT, ROLLBACK,
VETO_OBLIGATION, OBLIGATION_STATUS_CHANGE
```

Each entry carries `ledger_id`, `event_type`, `timestamp`,
`parent_event_id`, `payload_hash`, `payload`, `actor`, `version`.
Frozen dataclass + tuple-only `entries()` accessor + canonical JSON
hashing guarantee the append-only invariant. `verify_append_only(base)`
returns False if any prior entry was modified between snapshots.

## VetoToTestSynthesiser

Valid veto → obligation → pytest skeleton. Skeleton contains risk,
failure_case, proposed_test inside a docstring + a `pytest.fail()`
stub so the obligation surfaces in CI until a human implements it.
Status lifecycle: `OPEN → IMPLEMENTED → PASSED|FAILED` plus `WAIVED`
(requires non-empty waiver_rationale; refused otherwise). Status
transitions return a NEW obligation object; the original is never
mutated.

## Ledger-aware promotion

`promote_with_ledger` adds four completeness checks on top of v0.5's
`PromotionGate`:

1. PROPOSAL entry exists for `mutation_id`
2. EVALUATION entry exists for `mutation_id`
3. JURY_ROUND2 entry exists for `mutation_id`
4. every VETO_OBLIGATION linked to `mutation_id` has its latest
   OBLIGATION_STATUS_CHANGE in a terminal state

On success, appends SNAPSHOT + PROMOTION_DECISION entries.

## PathQualityMetrics

Deterministic raw counters: timeline_length, branch_opened_count,
guard_blocked_count, contradicts_count, merged_into_count,
hook_error_count. v0.6 ships no statistical layer; v0.7 will build
deltas on top.

## ProposalDraftBuilder

From a `ReflectionFinding` to a `MutationProposal` with
`requires_ratification=True`. `promote()` refuses unratified drafts.

## What v0.6 deliberately does NOT do

* no real LLM API calls
* no autonomous internet access
* no behaviour-effective config_delta
* no automatic promotion without a complete ledger
* no overwriting of any ledger entry
* no memory reads during a run
* no statistical significance tests
* no parallel-clone scheduler
