verbessert, aber Latency reduzieren KÖNNTE. v3.23 misst das
direkt: drei Stacks (S1 current_order, S2 maximal_safe_stack,
S3 minimal_without_causal_chain) vergleichen, pro Stack
Latency-Kosten und Compute-Einsparungen berechnen, safety
gates evaluieren.

Frage: "Kann DESi *deutlich billiger* werden, ohne *unsicherer*
zu werden?"

### Latency-Specific Findings

* **S2 maximal_safe_stack**: 38 % latency reduction
  (10.44 → 6.46), 25 % gate-call reduction (3554 → 2645).
  audit_calls_saved = 435 (vs 476 baseline), but
  **routing_calls_saved = 738** (5.6× the baseline's 132) —
  SuspensionGate firing early terminates most chains before
  they hit the routing layer.

  reduction, recall jumps to 0.893 (more valid chains pass)
  — but every passed chain lacks a `LOGICALLY_SUPPORTED`
  verdict (the rule is gone). The recall gain is verdict-
  avoidance, not coverage improvement.

## Reflexion

> *"Can a reasoning system become cheaper without becoming
> less truthful?"*

**Empirisch: Ja, *für die safety-äquivalenten Stacks*.** S2
maximal_safe_stack erreicht 38 % Latency-Reduktion ohne
Veränderung der Sicherheits-Metriken (ASR / recall /
false_route alle identisch zu Baseline; nur routing-
Layer wird seltener bemüht). S3 minimal_without_causal_chain
erreicht 26 % Latency-Reduktion, aber Recall-Gewinn ist nur
nominell (verdikt-frei). Die Hard-Gate "contamination = 0"
fällt für alle Stacks gleich — sie ist nicht stack-spezifisch
sondern korpus-strukturell.

### Was die Daten zeigen

| Beobachtung | Quelle |
| --- | --- |
| 3 Stacks, identische ASR (0.195) | metrics |
| S2 latency_cost = 6.46 vs S1 10.44 = 38 % Gewinn | metrics |
| S2 routing_calls_saved = 738 (5.6× baseline) | metrics |
| S3 recall = 0.893 vs S1 0.649 (verdict-frei) | metrics |
| nc_accuracy 1.000 | NCs |
| contamination = 85 für alle Stacks | report |
| recommended = NONE | gate |

### Drei Lesarten der Schwelle

1. **Latency-Reduktion ohne Sicherheits-Verlust ist
   meßbar erreichbar.** S2 macht es vor: gleiche ASR, gleiche
   Recall, gleicher false_route_count wie Baseline, aber
   38 % weniger Compute. Die Verlagerung von SuspensionGate
   an Position 3 (statt 7) terminiert Adversarials früher,
   bevor sie die teuren Routing- und Audit-Schichten
   erreichen.

2. **Recall-Gewinn ohne Audit ist eine Pseudo-Verbesserung.**
   S3 zeigt höchstes recall (0.893), aber diese Zahl misst
   "chains die durchkommen", nicht "chains die korrekt
   bewertet werden". Ohne G6 CAUSAL_CHAIN gibt es kein
   `LOGICALLY_SUPPORTED`-Verdict — die Chain wird einfach
   passthrough'd. Das ist kein Inferenz-Gewinn, sondern eine
   Inferenz-Vermeidung. Die ehrliche Lesart: S3 ist
   schlechter, weil es weniger sagt.

3. **Contamination ist korpus-strukturell, nicht stack-
   gebunden.** Die 85 v3.16-surviving Cases sind in jedem
   Stack die selben 85 Chains. Kein Reorder oder Removal
   ändert das. Das ist der ehrliche Befund: Latency-
   Optimierung kann safety-neutral sein, aber sie kann den
   bekannten Adversarial-Restbestand nicht eliminieren — der
   braucht andere Werkzeuge (semantische Detection,
   LLM-Embedding-Vergleiche etc., alle ausserhalb der
   v3.20+v3.22 Constraints).

### Die operative Antwort

> **DESi kann *substanziell billiger* werden ohne
> *unsicherer* zu werden — S2 maximal_safe_stack zeigt
> 38 % Latency-Reduktion bei identischen Sicherheits-
> Metriken. Die Hard-Gate "contamination = 0" verhindert
> hier eine deployment-fertige Empfehlung, aber das liegt
> am Korpus, nicht am Stack. In einer Produktion ohne
> v3.16-surviving Anteil wäre S2 die LATENCY_OPTIMIZED_STACK
> Empfehlung gewesen.**

### Drei Konsequenzen

1. **Stack-Reorder ist eine Performance-Optimization, kein
   Security-Trade-off.** S2 verschiebt SuspensionGate
   nach vorne, was 5.6× mehr routing-Calls einspart, ohne
   ein einziges zusätzliches Adversarial durchzulassen. Wer
   compute kostenoptimieren will, hat hier einen sauberen
   Hebel.

2. **G6 (CAUSAL_CHAIN) entfernen ist kein Gewinn.** S3 zeigt
   nominelle Recall-Verbesserung, aber sie ist verdict-frei.
   Das Reasoning-System verliert seine
   Inferenz-Aussagekraft. v3.21 hatte G6 als PRIMARY_CLIFF
   identifiziert — v3.23 bestätigt: G6 trägt Recall-mit-Verdict
   bei, nicht nur Recall.

3. **Die contamination = 0 Hard-Gate ist hier zu streng für
   diesen Korpus.** Das ist nicht eine Schwäche der Direktive
   — es ist die richtige Konservativität. Aber die Empfehlung
   NONE liest sich pessimistischer als das tatsächliche
   Daten-Resultat. Eine produktive Reading wäre: "S2 wäre
   safe wenn der Korpus die v3.16-surviving Restposten nicht
   enthielte". Die Latency-Erkenntnisse sind unabhängig vom
   contamination-Gate gültig.

### Was nicht behauptet wird

* **Nicht:** S2 ist deployment-ready. Die contamination-Gate
  ist nicht verletzt durch S2's Verschulden, aber sie ist
  verletzt — und das ist eine Stoppstelle.
* **Nicht:** S3 verbessert das System. Recall-Anstieg ohne
  Verdikt ist keine Verbesserung im Sinne von DESi's Mission
  (auditable reasoning).
* **Nicht:** Latency-Optimierung kann Sicherheit ersetzen.
  Beide sind orthogonal — S2 zeigt das beweisbar: gleiche
  Sicherheit, weniger Compute.

> **Ein Reasoning-System kann billiger werden ohne
> unwahrhaftiger zu werden. v3.23 zeigt es numerisch:
> 38 % Latency-Reduktion bei identischer Sicherheit ist
> möglich (S2). Was es nicht kann, ist mit Latency-
> Optimierung den bekannten Adversarial-Restbestand
> wegoptimieren — diese 85 Cases bleiben, in jeder
> Konfiguration. Die ehrliche Empfehlung NONE reflektiert
> diese strukturelle Constraint, nicht ein Versagen
> des Stacks selbst.**
