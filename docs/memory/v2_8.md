# v2.8 — Rule Patch Protocol

**Branch:** `feature/rule-patch-protocol`
**Modus:** architectural extraction — destilliert den v2.7-Prozess
in ein wiederverwendbares, deterministisches Protokoll für jeden
zukünftigen Regel-Patch. **Keine** bestehende Regel verändert,
**keine** neuen ClaimStates / Operatoren / BridgeKinds / Ledger-
Events / Tool-Typen / Mutationen / Benchmark-Cases.

## PatchPhase (A1)

Genau sieben Werte, geschlossen:

```
DISCOVERY            – v2.4/v2.5/v2.6 Artefakte vorhanden?
RISK_PROBE           – v2.6 Probe: safe_to_implement?
GUARD_SYNTHESIS      – ≥ 2 Guards, alle observable, keine case_id-Checks
IMPLEMENTATION       – declared touched_files existieren + erlaubte Roots
REGRESSION           – sechs Benchmark-Hashes identisch zur Baseline
REPLAY_VERIFICATION  – zwei vollständige Läufe → identische Hashes
COMPLETE             – terminales Sentinel; erreicht nur wenn alles bestand
```

`PHASE_ORDER` fixiert die Auswertungsreihenfolge. Der Orchestrator
bricht beim ersten fehlgeschlagenen Phase ab.

## RulePatchRecord (A2)

Elf Pflichtfelder per Direktive:

```
patch_id                — sha256 über PatchCandidate
target_rule             — die mutierte Inferenz-Regel
source_branch
phase                   — höchste erreichte PatchPhase
passed                  — phase == COMPLETE && alle outcomes ok
created_guards          — Guard-Namen
touched_files
benchmark_hash_before
benchmark_hash_after
replay_hash             — deterministischer Hash über Record (ohne Timestamp)
timestamp
```

`replay_hash` schließt `timestamp` und sich selbst aus, damit zwei
Aufrufe auf identischen Inputs **bit-identische** Records produzieren.

## Guard-Synthese-Validierung (A3)

Der `GUARD_SYNTHESIS`-Runner erzwingt drei Disziplinen:

1. **`len(guards) >= 2`** — sonst sofort
   `fail_reason = "missing_guards"`.
2. **`observable in ALLOWED_OBSERVABLES`** — eine geschlossene
   Menge {`rule_iteration_order`, `premise_kind`,
   `premise_text_substring`, `conclusion_text_substring`,
   `premise_token_graph`, `conclusion_token_overlap`,
   `premise_count`, `conclusion_kind`}. Alles andere wird
   abgelehnt.
3. **Forbidden tokens** in `observable`: `case_id`, `allowlist`,
   `whitelist` → automatisch abgelehnt.

Damit ist die "echte Inferenzregel vs. getarnte Allowlist"-Frage
aus v2.7 maschinell entscheidbar.

```

Ergebnis:

```
phase         : guard_synthesis

Der Orchestrator bricht **vor** REGRESSION und REPLAY_VERIFICATION
ab — das Protokoll lässt einen Patch ohne strukturelle Guards gar
nicht erst nahe an die Benchmarks heran.

## Replay (A7)

- Zwei Läufe von v2.7-Rekonstruktion: identischer `replay_hash`.
- Zwei Läufe von Fake-Patch: identischer `replay_hash`.
- v2.7 vs Fake: distinkte `replay_hash`-Werte.

Hard Invariant: **same patch input → same replay_hash** über zwei
Prozesse → durch die deterministische JSON-Kanonisierung und das
Ausschließen des Timestamps gewährleistet.

## Aufgabe 8 — Reflexion

> *"When does a rule patch become science instead of tinkering?"*

Aus den v1.0–v2.7-Daten ist die Antwort jetzt operativ erfassbar:

> **Ein Regel-Patch ist Wissenschaft, wenn er sich Schritt für
> Schritt durch ein deterministisches Protokoll führen lässt, das
> bei jeder Phase ein vorab definiertes Falsifikationskriterium
> auswertet und nur dann fortschreitet, wenn das Kriterium
> empirisch erfüllt ist.**

Konkret: die sieben Phasen schließen die folgenden
Tinkering-Failure-Modes systematisch aus:

| Tinkering-Modus | Welche Phase schließt ihn aus? |
| --- | --- |
| "Ich denke, das Problem ist X" ohne Daten | DISCOVERY — Artefakte müssen existieren |
| "Es sieht nicht so schlimm aus" | RISK_PROBE — quantifizierte FP-Reopen-Rate |
| "Ich füge einen schnellen Fix für Case N hinzu" | GUARD_SYNTHESIS — case_id verboten, Allowlist verboten |
| "Ich ändere noch dies hier mit" | IMPLEMENTATION — touched_files-Allowlist |
| "Es lief auf meinem Rechner" | REGRESSION — Hash-Identität auf sechs Achsen |
| "Es lief gerade eben" | REPLAY_VERIFICATION — zwei Läufe müssen bit-gleich sein |
| "Genug ist genug, mergen wir" | COMPLETE — terminales Sentinel, nur nach allen anderen |

Das v2.6 → v2.7-Workflow ist die erste empirische Bestätigung
dieses Protokolls. Vor v2.7 war jede DESi-Version entweder Bestand
(v1.x) oder reine Beobachtung (v2.0–v2.6). v2.7 war der erste
Patch, der das Protokoll *implizit* durchlief. v2.8 macht das
Protokoll **explizit, reproduzierbar und prüfbar** — die nächste
v2.x-Version (sollte sie kommen) hat keinen Ermessensspielraum
mehr: sie wird das gleiche Skelett durchlaufen oder als Tinkering
abgewiesen.

Die operative Definition von "Science" hier ist eng:
> **Ein Patch ist Wissenschaft, wenn das Falsifikationskriterium
> für jeden Schritt *vor* der Implementation festgelegt ist und
> *nach* der Implementation maschinell überprüft werden kann.**

Der Fake-Patch ohne Guards ist genau das Negativbeispiel: er
verspricht eine neue Regel, hat aber keine Falsifikations-
Bedingung, an der man messen könnte, ob die Regel nicht zu
breit feuert. v2.8 weist ihn binnen Millisekunden zurück. Vor
v2.8 hätte er womöglich gemerged werden können — heute nicht
mehr.
