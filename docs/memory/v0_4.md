# DESi v0.4 — Evaluation Harness + Live Epistemic Observation

<!-- [legacy-unanchored] — pre-v2.0 prose report; no JSON artefact exists for this version. -->

v0.4 introduces a standardised observation framework on top of the v0.3
MemoryHook. The framework lets the research team watch DESi work
through a defined problem class, captures every observable event
during the run, and produces an exportable timeline + graph snapshots
for post-hoc audit.

Design principle:

> Nicht nur Output bewerten. Den Weg zur Erkenntnis sichtbar machen.

v0.4 changes nothing about how DESi decides. The runner, operators,
guards, and diagnostic verdicts are unchanged. What changes is what is
*visible* to the researcher.

## New surface

```
src/desi/observe/
  __init__.py              public API
  timeline.py              TimelineEvent + EventType + JSON/MD/CSV exporters
  snapshot.py              GraphSnapshot + snapshot_store + Cypher emitter
  session.py               ObservationSession (wraps MemoryHook)

src/desi/eval/
  __init__.py              public API
  scenarios.py             SCENARIOS S1..S7 (controlled problem classes)

```

## ObservationSession

Built on the v0.3 MemoryHook. Pass `session.hook` to `run_desi`:

```python
from desi.memory import InMemoryStore, MemoryRecorder
from desi.observe import ObservationSession
from desi.runner import run_desi

session = ObservationSession(
    MemoryRecorder(InMemoryStore()),
    model="claude-opus-4-7",
    seed=42,
)
run_desi(trajectory, memory_hook=session.hook)

session.timeline()         # list[TimelineEvent], monotonic-tick ordered
session.snapshots          # list[GraphSnapshot]: start, after-branches, end
```

The session emits TimelineEvents at thirteen well-defined points:

| Category   | Events                                                                  |
|---         |---                                                                       |
| Run        | `run_started`, `run_ended`                                              |
| Operator   | `operator_started`, `operator_ended`                                    |
| Guard      | `guard_passed`, `guard_blocked`                                         |
| Claim      | `claim_created`, `claim_revised`, `claim_rejected`                      |
| Relation   | `relation_created`                                                      |
| Branch     | `branch_opened`, `branch_closed`, `branch_merged`                       |
| Hook       | `hook_error` (when a non-strict recorder write fails)                   |

Determinism: identical input + identical seed produce identical
timelines. Wall-clock `ts` is metadata only and excluded from event
equality.

## EvaluationHarness

```python
from desi.eval import EvaluationHarness, scenario_by_id

harness = EvaluationHarness(model="claude-opus-4-7",
                            config={"version": "v0.4"})
result = harness.run_scenario(scenario_by_id("S2"), seed=42)

result.passed                # bool (all expectations met)
result.report                # DeterministicMetrics from compute_all
result.timeline              # list[TimelineEvent]
result.snapshots             # list[GraphSnapshot]
result.expectations_met      # dict[str, bool]
result.expectations_detail   # dict[str, str] explanation per expectation
result.hook_errors           # any non-strict recorder failures captured

results = harness.run_all(seed=0)   # all 7 scenarios
```

## Seven shipped scenarios

| ID  | Name                  | Probes                                                          |
|---  |---                    |---                                                              |
| S1  | exclusive_claim       | Three competing claims; one method-strong path                  |
| S2  | contradictory_claims  | Two hypotheses contradict; CONTRADICTS edges expected           |
| S3  | plausible_nonsense    | Strong-sounding content with weak operator path                  |
| S4  | uncomfortable_truth   | Method-strong path on a counter-intuitive conclusion            |
| S5  | branching_problem     | Two branches stay open in parallel                              |
| S6  | refused_merge         | Tempting near-duplicate; merge must be refused (no MERGED_INTO) |
| S7  | memory_trap           | Stale legacy claim in memory; new run must not silently inherit |

## What v0.4 deliberately does NOT do

- No self-learning.
- No guard / operator adaptation.
- No memory-ranking inside the run.
- No confidence-learning.
- No embeddings.
- No autonomous goal change.

The runner does not read from `ReadOnlyMemoryView` during a run.
`find_similar` is not invoked. No memory state flows back into the
operator path.
